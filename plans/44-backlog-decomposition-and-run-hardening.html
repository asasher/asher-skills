Small skills, a navigable source tree, shared memory, and a hardened backlog

The current backlog skill owns reusable engineering techniques inside its private references and setup templates, the growing flat skills directory is becoming harder to navigate, and three production run retros plus their follow-on failure issues show that operational fixes trapped in prompts or Claude-only auto-memory are repeatedly rediscovered. Split stable disciplines into compact, deep modules with setup-owned playbooks; group sources by human-facing domain without changing their names; replace harness-private memory; then harden backlog’s remaining orchestration. Compactness is load-bearing: fewer tokens, less to read, and smaller units that can be understood correctly in isolation.

Setup is an owned reference, not another skill

Setup is a branch of the public skill’s command surface. The owning SKILL.md routes setup to an ordinary bundled reference such as reference/setup.md; no nested SKILL.md, hidden skill identity, or separate setup dispatcher is introduced. setup-asher-skills owns closure-wide installation and invokes each installed owner’s setup branch by name.

ConcernContract
DeclarationThe public root’s command table declares setup and points to reference/setup.md; required and optional sibling names remain canonical dependency declarations. A broken pointer is a packaging error. No setup command means the skill has no setup branch.
DiscoverySkill discovery reads only skills/<category>/<name>/SKILL.md. Setup references are ordinary bundled markdown reached by context pointers, so catalog, installer, and model metadata never mistake them for independent skills.
InvocationA user may invoke <skill> setup. For installation, setup-asher-skills resolves the selected skill’s required sibling closure, presents the total write plan, installs dependency-first, then invokes each installed owner’s setup branch dependency-first. It names the public skill; it never imports or duplicates the setup reference.
Ordering and cyclesRequired-sibling edges define a deterministic topological order; a required cycle is a hard pre-write error naming the full cycle. Optional siblings join only when installed or explicitly selected and cannot silently become required.

Stories

  • As a project maintainer, I want diagnosing-bugs independently installable, so that the same debugging discipline works without installing backlog.
  • As a backlog orchestrator, I want to assign a named skill to each delegated worker, so that child threads receive a bounded process without importing sibling files.
  • As a delegated worker, I want access to installed model-invoked skills even when the parent did not name one, so that I can reach useful ambient capabilities when the task reveals the need.
  • As a project being set up, I want each public skill’s setup command to route to its own bundled setup reference, so that configuration ownership follows capability ownership without inventing another skill.
  • As a human running several tickets, I want bounded waves and a durable status board, so that progress stays visible and a quota or session failure does not erase the run’s state.
  • As a resumed orchestrator, I want an audited resume based on durable evidence, so that I do not trust stale PIDs, dead watchers, or optimistic agent reports.
  • As a project using a live review gate, I want only a genuine human verdict to approve work, so that agents cannot self-approve or mistake an unpublished route for a valid gate.
  • As a verifier, I want setup to prove the real auth and drive-to-feature path, so that environment failures are identified before they are misdiagnosed as product defects.
  • As an agent that discovers a durable fact, I want a model-invoked memory skill to classify and store it in the right shared owner, so that knowledge is not trapped in one harness or duplicated everywhere.
  • As the same user on another machine or harness, I want cross-project memories available through the approved shared owner, so that recall and correction travel with me without assuming the storage decision in advance.
  • As a maintainer browsing a growing collection, I want skill sources grouped by their primary user job, so that I can find related skills without changing how any skill is invoked or installed.
  • As a Codex orchestrator, I want staffing to dispatch work to the Claude sibling harness through claude -p without --bare, so that either harness may delegate to the other while each direction remains independently fallible.
  • As a workflow author, I want to call a skill through a narrow interface instead of repeating its discipline, so that the caller stays short and has only one module boundary to understand.
  • As a skill reader, I want each runtime branch to load only the material it uses, so that token cost and human reading effort scale with the selected path rather than the whole system.
  • As a user running a long implementation, I want the system to notice shrinking usage headroom before a hard stop, so that it finishes durable work, adjusts staffing, or pauses cleanly instead of stranding a fleet.
  • As a returning orchestrator after a reset, I want a scheduled audited continuation, so that productive work resumes without asking me to reconstruct or manually restart the run.
  • As an upstream skill author, I want copied, adapted, and inspired work credited in the consuming skill’s README with preserved license obligations, so that decomposition never erases lineage.

Key decisions

  • Two independent axes. Invocation policy and delegation policy are recorded separately; “primitive” does not imply model-invoked, and “orchestrator” does not imply delegated execution.
  • Explicit and ambient composition coexist. A parent may name any installed skill in a child prompt; inside the child, only model-invoked skills are autonomously discoverable.
  • Setup is a reference branch, not a skill. A public skill routes its setup command to reference/setup.md, which creates/reconciles its playbooks. setup-asher-skills coordinates closure-wide invocation by public skill name and never imports another skill’s setup file.
  • diagnosing-bugs is the pilot extraction. It has the strongest independent invocation, smallest coupling surface, and clearest provenance. Backlog keeps the issue lifecycle around it, not the technique itself.
  • No big-bang decomposition. Other candidates leave backlog only in individually evaluated cuts; the end-to-end issue loop must stay green after each extraction.
  • Durable truth beats agent reports. Checkpoints, refs, tracker state, event logs, and verified process identity are authoritative for status and resume; a message claiming “watch armed” or “merged” is not.
  • Final-HEAD evidence remains invariant. Verification may retain a scoped stack/fixture lease, but human-facing evidence is finalized only after review convergence against the reviewed HEAD.
  • Retros are input, not a new mandatory ceremony. This roadmap consumes #44–49, #51–55, and #57 plus the named closed regression issues; it does not add a retrospective stage to every ticket.
  • Harness memory is learned from, migrated, then deprecated. The memory plan inventories every installed harness-specific memory surface, preserves useful behavior deliberately, and approves a reversible migration before setup disables writes or reads. Claude auto-memory is the first known target, not the only possible one; no useful entry is silently deleted.
  • Memory is routing, not a new dumping ground. Project invariants land in their existing repo instruction/playbook owner; active decisions and incidents land on the issue tracker; secrets remain in secret storage; ephemeral or stale facts are dropped; only genuinely cross-project knowledge enters the global store.
  • Memory storage defaults are approval-gated. A private git-backed store is the baseline candidate because it offers harness-neutral history, sync, and offline reads, but the dedicated memory plan must compare it with the real alternatives. Whatever wins is private where required and retrieved lazily rather than injected wholesale into every turn.
  • Categories are physical, not semantic namespaces. The source tree gains a shallow domain grouping inspired by Matt Pocock’s repository; skill names, dependency references, installed copies, and workspace names do not inherit the category path.
  • Group by primary user job. Categories describe why someone reaches for a skill—system, delivery, creative, thinking, personal—not whether it is primitive/orchestrator or user/model-invoked. Those remain independent metadata axes.
  • #48 and #49 land in the first production cut. Dispatch-time coordinator staffing uses facts already available from grooming. Sibling harness dispatch is symmetric while both directions work: Codex reaches Claude through claude -p without --bare; each direction has its own availability state and fallback.
  • Reachability and economics are compiled facts. Staffing records each harness→model route with its auth source, cost pool, allowance/credit, health check, and fallback. Vendor announcements never become capability until the route works end to end.
  • Compactness is a correctness tool, not aesthetic minification. The target is less material in each caller and branch, achieved by removing duplication/no-ops and deepening module boundaries; required completion criteria stay inside the owning skill even when they cost lines.
  • Deep modules expose four things. A caller needs the skill’s trigger/input, output, completion gate, and failure/escalation contract. The worker owns the full method. If the caller must understand the method to invoke the skill safely, the boundary is too shallow.
  • Load traces are the token budget. Compactness is measured per real path—always-loaded metadata, composer, selected worker, selected references, setup-only—not by total repository size or an arbitrary file-length cap.
  • The graph is generated architecture. Canonical dependency/setup declarations drive the catalog and checked graph data; the rendered plan visual explains that graph but is not a second hand-maintained source of truth.
  • Capacity is a separate deep module. Usage observation and reset interpretation are reusable across backlog, skill-loop, and ad-hoc delegated work. capacity produces a snapshot; staffing routes from it; callers decide how to checkpoint and resume.
  • Unknown is not healthy. Every pool observation carries source, time, and confidence. When a harness exposes no reliable meter, capacity reports unknown and callers reduce commitment; they never invent a remaining percentage or reset timestamp.
  • Optimize completed durable units, not concurrency. The scheduler compares reliable headroom to the predicted cost of reaching the next checkpoint. It prefers finishing near-complete work, shrinks waves before exhaustion, and never routes below capability or taste floors just to consume another pool.
  • Hard limits become scheduled pauses. On exhaustion, the active thread writes a durable checkpoint and status, releases scarce resources, records the earliest safe continuation time, and installs a harness-bound wakeup. Resume begins with the same durable-state audit used after a fleet kill.
  • Capacity is machine/account-wide. Concurrent projects share usage pools, so observations and active reservations live in an atomic machine-level state surface rather than isolated repo playbooks; project policy may cap consumption further but cannot create headroom.
  • Staff before spawning. The parent that owns a delegation consults staffing with the task facts and capacity before creating the child. For backlog’s first delegation, that parent is the run thread and the selected role is an issue coordinator—not the session orchestrator by default.
  • Skill invocation is not an agent boundary. A coordinator or delegated worker may invoke model-invoked skills in its current context. A new agent exists only at an explicit dispatch boundary, which records parent, purpose, selected route, thread id, capacity pool, durable return contract, and escalation successor.
  • Lineage is part of the skill’s human contract. When another skill or project materially shaped a skill, its README states whether the relationship is copied, adapted, inspired, or internally extracted; names and links the source; describes local changes; and carries every license/notice obligation without loading that history into runtime instructions.
  • Deprecation proceeds by parity gates, not a flag flip. Harness-private memory moves through inventory, approved routing, migration, shadow comparison, write disablement, read parity, and only then read disablement/removal. Each phase has rollback and preserves an accepted snapshot until the next gate passes.
  • This is a roadmap with separately authorized tranches. Approval settles the architecture, order, and child-plan gates; it does not authorize one giant change, memory migration/disablement, capacity deployment, or every later extraction. Each tranche can land and roll back independently.
  • Events are canonical; boards are projections. Each parent writes structured spawn/return events to the platform-bound durable surface. Status can be rebuilt after the run thread exits, so monitoring convenience never becomes a single point of truth.
  • Greenfield instructions are harness-neutral first. Setup creates AGENTS.md as the canonical project instruction file when neither instruction file exists; when Claude Code is installed it also creates the minimal CLAUDE.md @AGENTS.md import plus Claude-only deltas. Existing deliberate layouts are reconciled, never replaced.
  • Setup promises only owner-routed behavior. Review-loop’s public setup command gains an ordinary setup reference for presentation configuration. Staffing’s setup reference preserves an existing global base and writes a missing base only with explicit consent; otherwise it reconciles the project delta. Installer prose names those exact degradations.
  • Mechanism claims remain hypotheses until falsified. Plans label unfamiliar-library/runtime assumptions and their cheapest red-capable probe. Implement validates the riskiest first; verify and review open every claimed test/artifact before treating it as coverage.
  • Every criterion declares how it can be exercised. Entities, account/tenant needs, scale, fixture lifetime, and observation mechanism are part of the criterion. Infeasible requirements are rewritten or their substitute is approved at the plan gate, never improvised by verification.
  • Self-evident cosmetic bugs get a guarded direct-fix lane. A human report plus an observable visual/copy defect and named styling/copy cause may go straight to fix and visual verification; the first contradictory observation falls back to diagnosing-bugs.
  • Evidence validity follows HEAD, not ceremony. Styling-only verification captures may double as evidence only when review changes no product code and the reviewer explicitly confirms no recapture is required. Any code/fixture/environment change invalidates them; ordinary final-HEAD capture remains the default.
  • Review serving and verdict delivery have separate durable owners. The HTTP server launches outside the issuing exec session with closed inherited descriptors, state-scoped logs, atomic PID/port metadata, health verification, idle/explicit teardown, and stale-registry repair. Watcher loss cannot kill serving; server loss is surfaced before awaiting.
  • Behavior rulings return to lifecycle authority. A Reviewer that finds a genuine product-semantics question stops and sends it to the issue coordinator; the coordinator resolves within the approved plan or escalates to the session orchestrator/human, then sends a ruling to the Fixer, requires targeted re-verification, and returns the new HEAD to the same Reviewer.
  • Native dependency edges and final handoffs are durable platform data. Setup probes native dependency read/write verbs when available; run uses those edges for waves. On terminal handoff it writes both tracker summary and a cross-harness file snapshot under the shared run administrative root, including environment state and cleanup debts.
  • Correct behavior earns a ruthless compaction pass. Refactoring is not complete when the new graph merely works. Every changed skill is pruned for duplication, no-ops, sediment, and unnecessary always-loaded detail, then re-proven against the locked behavior. Compactness serves the agent through lower token load and the human through less, clearer material to read.
  • No arbitrary line-count target. The budget is situated load and comprehension, not a tiny file for its own sake. Required gates stay; branch-only/setup-only detail moves behind precise context pointers; a shorter file that hides authority or changes behavior fails.

Risks and mitigations
  • Dependency sprawl: extraction can turn a readable workflow into a catalog hunt. Mitigation: extract only independent invocation or sequence boundaries, and keep backlog’s lifecycle visible in one issue-loop contract.
  • Duplicate authority: copied technique may survive in backlog and the new skill. Mitigation: one source of truth; backlog retains only invocation and handoff language.
  • Setup conflict: multiple public skills’ setup references may edit shared playbooks such as environment.md. Mitigation: section ownership plus a reconciler that preserves unowned sections and deliberate project deltas.
  • Oversized change: retro hardening and decomposition could become one unsafe rewrite. Mitigation: ordered slices, each leaving the end-to-end issue loop usable and independently evaluated.
  • Harness limitations: some dead-wait behavior ultimately belongs to the harness. Mitigation: encode durable fallbacks now and file harness changes separately; do not make correctness depend on a future harness fix.
  • Global-memory sprawl or leakage: a shared store can become a second unbounded memory pile or receive secrets. Mitigation: strict routing, approved access controls and sync surface, provenance/expiry metadata, no secrets, lazy retrieval, and periodic reconciliation.
  • Taxonomy churn: categories can become a recurring naming debate. Mitigation: shallow primary-job groups, one home per skill, no runtime meaning, and category changes treated as rare navigation refactors.
  • Sibling-route volatility: a currently working dispatch direction can later fail because a harness changes. Mitigation: treat reachability as per-direction staffing state, fail closed on the broken route, preserve the other direction, and repair the graph when the failure is encountered rather than embedding a vendor-policy monitor.
  • Over-pruning: token pressure can delete a caveat or gate that makes a skill safe. Mitigation: prune duplication and no-ops first, preserve observable completion/failure criteria, and require existing behavioral probes to pass after every compacting edit.
  • Shallow-module proliferation: many tiny skills can cost more navigation than they save. Mitigation: split only at independent invocation or sequence boundaries and require each extracted skill to hide meaningful complexity behind a narrow interface.
  • Bad telemetry: stale or partial vendor meters can trigger the wrong staffing decision. Mitigation: source/time/confidence on every observation, explicit unknown state, and conservative commitments when data is weak.
  • Monitoring tax: a watcher can consume the usage it is trying to preserve. Mitigation: cheap read-only probes at stage transitions, no capable parked model, and event/reset-driven wakeups.
  • Wakeup failure: a scheduled continuation may never fire. Mitigation: durable not_before state, an idempotent wake job, and ordinary next invocation detecting overdue resumable work.
Out of scope
  • Extracting every backlog reference merely because it is short.
  • Adding a mandatory retrospective stage to every issue lifecycle.
  • Rewriting tracker bindings or replacing the existing issue/PR role vocabulary.
  • Changing the product repositories whose runs produced #44, #46, and #47; this plan changes the reusable skills and setup baselines.
  • Disabling authored instruction surfaces such as project AGENTS.md/CLAUDE.md; the plan deprecates agent-written harness memory only after parity, not deliberate versioned instructions.
  • Auto-loading the entire global memory store into every context; retrieval is skill-driven and scoped to the active need.
  • Renaming skills or introducing category-qualified invocation/dependency names as part of the source-tree migration.
  • Adding --bare to the Codex→Claude sibling harness dispatch path, or embedding an Anthropic-policy monitor in the skill; observed route behavior is the operational signal.
  • Bypassing vendor usage limits, purchasing or enabling additional paid capacity automatically, or routing work below its capability/taste floor merely to keep a run active.
