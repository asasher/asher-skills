User stories

  • As a skill maintainer, I want backlog to reference staffing/review/plan/prototype by plain name, so that improving a primitive improves every caller without forking machinery.
  • As an issue thread running the enhancement branch, I want to invoke the plan skill for the gated plan and keep the commit-and-implement tail in backlog, so that the dev workflow is unbroken.
  • As a reader of the slimmed backlog, I want every bundled-reference and sibling pointer to resolve to something that exists, so that no step dead-ends on a deleted file.
  • As someone running backlog setup on a fresh repo, I want setup to ensure the four siblings are present rather than install their sections itself, so that each capability is provisioned by its owner.
  • As a maintainer reading SKILL.md, I want the command table to be a dev-scoped map that declares the three-part dependency surface, so that the composition is legible.

Implementation decisions

  • Delete-not-stub the four references + review scripts + extracted templates. The siblings carry the full contract; leaving stubs would re-introduce the fork. Removal is verified by absence (ac-1) and by zero dangling pointers (ac-3).
  • Roles vocabulary moves to staffing. orchestrator/builder/checker/floor + fallback ladder now resolve against the staffing skill by name; kept dev refs point to it, not a bundled file. (plan/prototype already phrase staffing this way.)
  • The enhancement dev-tail stays in backlog. The plan skill stops at approval ("committing the plan and starting the work are the caller's concern"), so backlog's issue-loop owns committing the approved plan to the work branch and the tracker posterity digest. Folded into issue-loop.md step 3 — no new reference file (minimalism); alternative flagged in Q4.
  • templates/environment.md § Model staffing and § Presenting become sibling-owned. § Model staffing is removed (staffing installs the roster via its own two-layer install); § Presenting reduces to a pointer noting review-loop owns the surface config (review-loop's contract already resolves to environment.md § Presenting). Exact ownership of that section couples to review-loop's setup — Q3.
  • Scope = the skill, not this repo's already-installed playbooks. #6 rewrites the templates and setup.md instructions; it does not re-run setup to slim this repo's live docs/agents/planning.md, prototyping.md, environment.md, AGENTS.md § Picking models, or CLAUDE.md. That reconcile-on-existing-repo pass is the separate installer / setup-re-run work — Q2.
  • Keep the .agents/skills/backlog/ mirror byte-identical to the skills/backlog/ source via mechanical copy, preserving the current invariant (diff -rq → 0). npx skills add is not used (it would pull from the parent checkout, which this worktree must not touch) — Q1.
  • No prototype needed (plan gate 2 clears). The composition boundary is settled on paper by reading the four sibling contracts; there is no unresolved state model or UI to see.

Risks
  • Self-modification: backlog is editing itself, but only in this worktree — the live orchestrator reads the primary checkout and is unaffected until #6 merges.
  • A rewired pointer that names a sibling the repo hasn't installed degrades silently. Mitigation: ac-8's resolution sweep + the dependency-surface declaration (ac-5) make the requirement explicit; setup (ac-7) ensures presence.
  • Boundary drift: if review-loop/staffing change their playbook-install shape later, dec-4's environment.md pointer could go stale — Q3 keeps that ownership explicit.
Out of scope
  • Re-running backlog setup against this repo to slim its already-installed docs/agents/AGENTS.md/CLAUDE.md (dec-5 / Q2) — separate installer work.
  • Installing the four siblings into this repo's .agents/skills/ runtime mirror so its own dogfooded backlog resolves them (an install/setup action, not a skill-content edit).
  • Any change to the four sibling skills themselves — they ship complete.
  • The goodwork review-surface migration (tracked under its own issue).
