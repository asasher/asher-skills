Setup asks the work domain, installs a domain baseline pack

A repo tracking non-code work (a book, a research effort, ops) runs backlog setup and silently gets test-first TDD playbooks — setup never asks what kind of work this is.

#32plan v1enables #35

The shape

The flat templates/ splits into a shared common/ plus per-domain packs; setup asks the domain once and installs common overlaid by the pack.

Stories

  • As the owner of a non-code repo (a book, research, ops), I want setup to ask what kind of work I track, so that my playbooks aren't TDD boilerplate that misfits every issue.
  • As the owner of an existing code repo, I want a setup re-run to change nothing — my install reconciles as software by default, so nothing regresses.
  • As the author of #35's writing pack, I want a templates/writing/ slot with a defined overlay rule, so the pack is a drop-in directory, not a re-architecture.
  • As a setup run for an unshipped domain, I want a flagged, reported degrade rather than a silent code-flavored install or a missing playbook that halts the loop.
  • As the issue-loop orchestration, I want to stay untouched — domain flavor lives in playbooks, never in references.

Key decisions

  • Common/pack split, not full per-domain sets. Shared contracts (bindings, label policy, audit slots) keep one canonical copy; a domain pack carries only the step playbooks that differ. Forced by #35's economics: a writing pack should be ~7 small files, not a fork of all 11.
  • The 4/7 classification. Common: backlog-policy, platform, environment, change-description — role bindings and setup-filled audit slots, no work technique. Pack: implementing, verifying, evidence, diagnosing-bugs, refactoring, change-reviewer, change-fixer — each inlines a technique (test-first, check discovery, screenshot evidence, code-review bar) that is what “code flavor” means.
  • Overlay rule: pack shadows common by filename. Install = common/ ⊕ templates/<domain>/, same-name pack file wins. Costs nothing today (no overlaps ship) and lets a future pack re-flavor a shared default without touching the structure.
  • The domain is recorded in the installed backlog-policy.md, a new § Work domain the common template ships as a slot. It is policy about what kind of work the tracker holds, so it lives with the label policy; setup re-runs read it, and absence means software — legacy installs reconcile unchanged.
  • Missing-pack degrade: flagged software baselines + a reported gap. The loop hard-stops on a missing playbook, so blanking the slots would brick every reference; installing software baselines silently would masquerade as fit. Each degraded file gets a header flag naming it a code-flavored stand-in to tailor; the setup report names the gap.
  • software/ is a verbatim move. The 7 step playbooks move with git mv, byte-identical; the only template content change anywhere is the § Work domain slot added to common/backlog-policy.md. This is the no-regression guarantee, checkable as pure renames in the diff.
  • The domain question opens setup's scaffold step (step 2) rather than becoming a new numbered step — the ordering guarantee (asked before any scaffolding) is stated in the criterion, and the reference avoids renumbering churn across its cross-references.
  • general is an option, not a pack. No shipped pack today, so it takes the dec-5 degrade path deliberately; authoring a genuinely neutral pack is its own future issue, as the writing pack is #35.

Definition of done

  • skills/backlog/templates/ contains exactly common/ (backlog-policy, platform, environment, change-description) and software/ (implementing, verifying, evidence, diagnosing-bugs, refactoring, change-reviewer, change-fixer); no flat templates/*.md file remains.
  • The diff against main shows all 11 templates as pure renames — byte-identical content — except common/backlog-policy.md, whose only change is the added § Work domain slot.
  • reference/setup.md asks the work-domain question before any playbook is scaffolded, with exactly the five options software / writing / research / ops / general, defaulting from repo evidence (confirm, not interrogate) and to software when evidence is ambiguous.
  • reference/setup.md states the install rule: the scaffold set is templates/common/ overlaid by templates/<domain>/, a same-name pack file shadowing the common file.
  • reference/setup.md states the missing-pack degrade: a domain with no shipped pack installs the software step baselines, each flagged in its installed header as a code-flavored stand-in to tailor, with the gap named in the setup report — no required playbook is ever left uninstalled.
  • reference/setup.md's re-run path reconciles against the work domain recorded in the installed backlog-policy.md, defaulting to software when no domain is recorded.
  • templates/common/backlog-policy.md ships a § Work domain slot setup fills, and this repo's own docs/agents/backlog-policy.md records software in it.
  • SKILL.md, README.md, and reference/run.md's preflight describe the common/ + <domain>/ layout; grep across skills/backlog/ finds no stale flat-templates/*.md phrasing.
  • Every bundled reference except setup.md and run.md is byte-identical to main, and the skill keeps its name — orchestration untouched, no rename.
  • Dual-executor situated probes for the domain path (new scenarios in skills/backlog/evals/probes.md: fresh writing-repo install hits the question + degrade; fresh code-repo install defaults to software; legacy re-run with no recorded domain reconciles as software) pass against an answer key written before any runs.
  • The installed copy .agents/skills/backlog/ is refreshed from source and content-matches skills/backlog/.

How it will be checked

Static criteria (ac-1, ac-2, ac-7–ac-9, ac-11) confirm at the repo seam: git diff --find-renames, directory listings, and greps on the branch. Behavioral criteria (ac-3–ac-6, ac-10) confirm at the probe seam per docs/agents/probe-evals.md: situated dry-run scenarios driven through both deployment executors (a Claude subagent in-session and gpt-5.6-sol via codex exec), graded pass/fail against a pre-written key — this repo's recorded stand-in for “drive the app” (environment.md § Seed data). Deliberately unchecked: a live end-to-end backlog setup against a real second repo — no fixture repo is in scope, and the probes exercise the same decision points at a fraction of the cost.

What's required

  • Probe transcripts from both executors plus a per-probe verdict table against the answer key (the evidence.md contract for this repo).
  • Terminal transcripts of the rename-diff and grep checks backing the static criteria.
Risks
  • Legacy installs (playbooks written before domain recording) could reconcile against the wrong pack — mitigated by the absent-→-software default (ac-6), which makes every pre-existing install resolve exactly as today.
  • The 4/7 classification could prove wrong for some future domain (e.g. a pack wanting its own environment.md) — mitigated by the overlay rule (dec-3): a pack may shadow any common file, so a misclassification is recoverable without restructuring.
  • Source and installed copy drifting mid-review — mitigated by refreshing the install in the same PR (ac-11) and re-running the refresh after any fixer commit that touches the source.
  • Other repos' installs go stale relative to the renamed layout — acceptable: installed playbooks are flat files under docs/agents/ and don't reference template paths; their next setup re-run reconciles them (recorded follow-up, not this PR).
Out of scope
  • #35's writing pack — this issue lands the slot it drops into, not the pack.
  • Authoring research / ops / general packs — each is its own future issue; until then those domains take the flagged degrade path.
  • Any rename of the skill or re-architecture of groom / run / issue-loop — the issue explicitly keeps orchestration untouched.
  • Re-running setup on other repos that have backlog installed — their reconcile happens in their own sessions.
