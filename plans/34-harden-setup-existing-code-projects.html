Harden backlog setup for existing non-trivial code projects

The loop was proven on a greenfield repo (no CI, no app, no tests) — three setup audits pass trivially here and would produce nothing usable when pointed at a real inherited codebase.

#34enhancementplan v1

The core problem

Setup already names all three concerns. But each audit has a trivial-pass path a greenfield repo slips through — "read a command catalog if present", "shared singletons: or none", "map each role where one fits". The hardening turns each audit from a mention into one that discovers concrete artifacts from a real repo and records them where the loop reads them. No new subskills; the change is to the audit contract (setup.md, worktree-isolation.md) and the playbook templates those audits write into.

Stories

  • As an operator running backlog setup on an inherited app, I want the audit to discover and verify the repo's real test/lint/typecheck/build commands and its CI merge-gate, so verify runs what actually gates the code and the PR can't merge on red CI.
  • As an operator, I want setup to record how to boot the stack and drive to a state that exercises a feature, so verify drives the real app and evidence is real screenshots, not agent-authored probe transcripts.
  • As an operator, I want the isolation audit to enumerate every shared singleton (DB, ports, node_modules, build caches, test DBs) with a real parallel/serial verdict, so run doesn't fan out worktrees that corrupt each other.
  • As an operator inheriting a large tracker with an arbitrary label taxonomy and non-GitHub dependency links, I want setup to map those labels to roles and bind the native dependency relation, and groom to triage the backlog at scale without re-litigating settled issues.

Key decisions

  • Harden audits, add no subskills. The three areas already have homes in the loop; the fix is to strengthen the audit contract and the playbook slots it fills, so the existing verify/run/groom/PR steps read richer artifacts unchanged.
  • Discovery is verify-by-running, not asking. "Binding to reality" means setup enumerates candidate commands from real config files and executes each, recording only those that ran — the same discipline the platform bindings already use. A greenfield repo yields an empty set honestly instead of passing on a guess.
  • The shared-singleton list is a first-class recorded artifact, not a verdict footnote. run and verify read the enumerated list; the verdict is derived from it and names the resource forcing serialization.
  • CI-green is a merge precondition, distinct from local checks. verify's local check gate proves the change; the host CI status gate (discovered in area 1) blocks the merge. Both are recorded; neither substitutes for the other.
  • Non-GitHub dependency relations bind as a tracker verb, written as a prose contract (Jira/Linear "blocks/blocked-by"). Verb commands are only verified against a live tracker at a real setup run — the template ships the contract, not fabricated commands.
  • Template contract change ⇒ version bump + migration. Bump templates/VERSION and add a migrations.md entry so setup's reconcile (step 2) upgrades an older install section-by-section, preserving repo-authored divergence.
  • Groom scale lives in groom.md, scoped to a first-groom affordance (batching + dedupe-without-re-litigating) — not a rewrite of grooming. Flagged below as the one item straddling the "setup + templates" boundary.

Definition of done

Area 1 — verify/evidence binding to reality

  • Check discovery + verify. setup.md step 5 directs enumerating check commands from real sources (package scripts, Makefile/Justfile, CI workflow files, pyproject/tox/noxfile, cargo/go/gradle) and running each to confirm it works, recording only commands that ran; verifying.md distinguishes discovered-and-verified checks from unverified guesses.
  • CI merge-gate discovered. Setup identifies the exact check set the host CI runs to gate merge, and verifying.md records it as the named CI gate (separate from the local narrowest-first checks).
  • Drive-to-feature state recorded. Setup step 6 (and environment.md § Seed / § Driving) records how to boot the stack and reach a state that exercises a feature — the entry/navigation path — as an explicit audit output, not folded into seed.
  • Real-app evidence over probes. evidence.md states that for a real running app, evidence is real check/test output plus screenshots driven through the app, and names agent-driven probe transcripts as a greenfield-only fallback for repos with no runnable surface.

Area 2 — isolation verdict with real shared singletons

  • Local-shared-resource probes. worktree-isolation.md audit probes include local collisions beyond containers/ports/cloud — shared node_modules/install dirs, build caches (.next/dist/target/build/__pycache__/.gradle), shared test databases, and state/lock dirs.
  • Enumerated singleton list. Setup step 4 + worktree-isolation.md require producing an explicit shared-singleton list — each resource with its collision mode and whether it's locally isolatable — recorded in environment.md § Worktree isolation, whose template slot is a list, not a one-line "or none".
  • Verdict as hard constraint. The docs state that when un-isolated real singletons exist, serialize-verification is a hard constraint that names the forcing resources (not a policy preference), and that a naive parallel fan-out would corrupt shared state.

Area 3 — label mapping, scale, non-GitHub deps, CI merge

  • Arbitrary-taxonomy mapping. setup.md step 3 handles a large pre-existing label taxonomy — mapping many labels to the readiness/work-type roles, resolving overlaps/ambiguity, leaving unrelated labels neutral, and reusing existing labels rather than creating duplicates.
  • Groom at scale. groom.md gains an affordance for a large pre-existing backlog: batching/prioritizing and de-duplicating without re-litigating already-settled issues, so a first groom of an inherited backlog is tractable.
  • Non-GitHub dependency relations. platform.md (tracker verbs + § Custom bindings) and backlog-policy.md § Dependencies bind a tracker's native dependency relation (e.g. Jira/Linear blocks/blocked-by) as a first-class verb run reads, stated per-binding.
  • CI-green merge gate. pr.md and verifying.md record the host CI status as a merge precondition — the PR is not considered mergeable until CI is green — distinct from the local check gate.

Meta

  • Versioned + migratable. templates/VERSION is bumped and migrations.md gains an entry describing each contract change and how reconcile applies it without deleting repo-authored content; every changed playbook template's stamp updates to the new version.
  • Internally consistent. The reference and template edits cross-reference correctly (setup writes exactly the slots the templates expose; verify/run/evidence/groom read exactly what setup records), with no dangling pointer to a removed or renamed slot.

How it will be checked

This is a docs/contract change with no runtime, so the criteria are confirmed by text review against each named file plus a situated dry-run probe (this repo's stateful-skill eval pattern): give an executor model a synthetic inherited-repo scenario — a JS/TS app with package.json scripts, a .github/workflows CI gate, a shared Postgres + node_modules, an arbitrary Jira-style label set — and confirm a setup dry-run produces the three hardened artifacts (verified checks + CI gate, enumerated singleton list + serialize verdict, label map + dependency binding). Left deliberately unchecked: an end-to-end setup against a live real repo — this skills repo has no CI/app/tests, so real validation is deferred to a first real-project run (see risks).

What's required on completion

  • The diff across the nine files + VERSION/migrations, each criterion mapped to the passage that satisfies it.
  • A dry-run probe transcript (executor model against the synthetic inherited-repo scenario) showing the three hardened artifacts emerge — the evidence surrogate for a repo that can't be set up against itself.
Risks & mitigations
  • Can't validate end-to-end here. asher-skills has no CI/app/tests, so the hardened audits can't run against it. Mitigation: dry-run probe against a synthetic scenario now; flag first-real-project validation as follow-up.
  • Jira/Linear verbs written blind. No live non-GitHub tracker to exercise. Mitigation: ship the dependency-relation binding as a prose contract with explicit "verify at setup time" framing (matching how custom bindings already work), not fabricated commands.
  • Scope breadth. Nine files touched. Mitigation: every edit follows one spine (audit → recorded slot → consumer); no new machinery, so the blast radius is prose contracts the loop already reads.
  • Prior art overlap. #27 ("harden setup audit") touched adjacent ground. Mitigation: this plan is written against current HEAD; edits layer on what #27 left, not a revert.
Out of scope
  • Writing isolation-scaffold code. worktree-isolation.md already describes the scaffold pattern; this hardens the audit only. Actual per-repo scaffolding stays a setup-time, approval-gated action.
  • Non-code / domain-pack support. The issue says it rides on these same setup improvements; deferred to its own track.
  • An end-to-end run against a real inherited repo. Deferred to the first real-project setup; this issue delivers the hardened contract + a dry-run probe.
  • Rewriting groom. Only a scale affordance is added; the grooming flow itself is unchanged.
