to-sprites — keyed sheets → transparent assets + manifest

A generated sprite sheet is a single flat-background image; using its elements on the web or in an engine means hand-slicing, keying, and bookkeeping every frame's coordinates.

What ships

A standalone, Codex-first skill skills/to-sprites/ whose value is a real runnable Python script: feed it a sprite sheet, it removes a flat chroma key, slices the sheet, trims each element, exports transparent PNG/WebP, and writes a spritesheet.json manifest — with an optional contact-sheet preview for QA. An optional front stage lets the starting point be a text prompt, generating the source sheet via the sibling codex-imagegen skill before the pipeline runs.

Optional generate-source stage

The starting point can be a prompt instead of a file. When --generate "<subject>" is passed, to-sprites composes the sibling codex-imagegen skill by plain name (no file import) to produce the source sheet, then feeds it into the unchanged key→slice→export→manifest pipeline.

Extraction pipeline

One deterministic pass over the sheet; each stage is a flag-gated step in extract_spritesheet.py. Color = stage family.

Skill layout

Modeled on shipped skills/staffing/ and skills/review-loop/; Codex-first (ships agents/openai.yaml). Keying lineage traces to the sibling codex-imagegen skill but is re-implemented here (cross-skill script imports are disallowed; codex-imagegen keys only magenta/green). codex-imagegen is also composed by name for the optional generate-source stage.

Manifest schema — spritesheet.json

One object per sheet; elements[] carries every field the issue lists. sheet_rect/frame are sheet coordinates (for CSS-sprite / atlas use); the exported PNG is the trimmed content.

{
  "source": "in/iso-4x4.png",
  "sheet": { "width": 1024, "height": 1024 },
  "slicing": { "mode": "grid", "cols": 4, "rows": 4,
               "tile": [256,256], "margin": [0,0], "spacing": [0,0] },
  "key": { "color": "#00FF00", "method": "border-sample" },
  "elements": [{
    "name": "tile_01_grass", "index": 0, "row": 0, "col": 0,
    "sheet_rect":     { "x":0, "y":0, "w":256, "h":256 },   // original cell
    "trimmed_bounds": { "x":18, "y":40, "w":220, "h":150 }, // content bbox in sheet
    "asset": "assets/tile_01_grass.png", "size": [220,150],
    "anchor": [110,150], "pivot": [0.5,1.0],                // bottom-center default
    "css":   { "background-position": "-18px -40px", "width":220, "height":150 },
    "frame": { "x":18, "y":40, "w":220, "h":150 }           // engine/atlas frame
  }]
}

User stories

  • As a game-asset maker, I want a keyed 4×4 iso tile sheet sliced into 16 named transparent PNGs, so that I can drop tiles straight into an engine.
  • As a web dev, I want a manifest with CSS background-position per element, so that I can use the sheet as a CSS sprite without measuring pixels by hand.
  • As a prototyper, I want the background auto-detected from the sheet border, so that I don't have to know the exact key hex.
  • As a user with an irregular packed sheet, I want connected-component slicing, so that elements without a fixed grid still come out as separate assets.
  • As a careful user, I want the original never overwritten and existing outputs refused unless I force it, so that a re-run can't destroy work.
  • As a QA reviewer, I want a contact-sheet preview and a validation report, so that I can spot empty cells, key remnants, or a wrong element count at a glance — without a browser.
  • As a Codex user, I want the skill invocable from the Codex CLI with a documented entrypoint, so that gpt-5.5 can run the extraction for me.
  • As a user starting from nothing, I want to hand the skill a text prompt and have it generate the source sheet (via codex-imagegen) before extracting, so that prompt→assets is one flow.

Implementation decisions

  • Runtime: Python 3 + numpy + Pillow only (both present on this machine: numpy 2.4.4, Pillow 12.2.0). No scipy — matches the codex-imagegen dep floor. Documented as pip install numpy pillow.
  • Connected components are labeled in-house (numpy 4-connectivity two-pass union-find) since scipy is absent; adequate for sprite-sheet-scale images.
  • Keying is generalized to an arbitrary flat key: --key auto (dominant border color by sampling the sheet's border ring), --key #RRGGBB (supplied), or --key none. Euclidean RGB distance with hi/lo alpha ramp + despill, re-implemented from codex-imagegen's magenta/green keyer.
  • Slicing has two modes: grid (cols×rows or tile size + margin + spacing) and components (label opaque blobs post-key). Grid handles the 4×4 iso fixture; both feed the same trim/name/export/manifest tail.
  • Naming: a provided list (--names a,b,… or --names file) maps row-major; otherwise stable auto names tile_NN, or tile_NN_<name> when a list is given (e.g. tile_01_grass.png). Ordering is deterministic across runs.
  • Non-destructive: source is opened read-only; outputs go to --out (default a new sibling dir); the writer refuses to clobber existing files unless --force. The source is never a write target.
  • Export: --format png|webp|both, all with alpha. Anchor/pivot default bottom-center (iso-friendly), overridable.
  • Entrypoint shape: one extract_spritesheet.py exposing both a CLI (--help documents all flags) and an importable extract(...) -> manifest dict so probes/tests assert on the returned manifest, not stdout.
  • Test fixture is generated deterministically by evals/make_fixture.py (Pillow draws 16 distinct iso diamonds on a flat green key) and committed — offline and reproducible, since headless-Chrome and live image-gen are unavailable/nondeterministic here. Real generated sheets remain the production input; the fixture only anchors verification.
  • Validation (--validate, with --expect N) checks alpha corners, empty cells, residual key-color remnants, and asset count; exits nonzero on failure so it doubles as the verify seam.
  • Optional generate-source (--generate "<subject>") composes the sibling codex-imagegen by plain name — it shells to that skill's scripts/codex_imagegen.py --subject … --key … --out <tmp sheet>, then runs the normal pipeline on the result. No file import; codex-imagegen is declared an optional sibling in the SKILL.md three-part dependency surface (bundled references · project playbooks · optional sibling: codex-imagegen). Absent/unreachable codex-imagegen degrades cleanly: the flag errors with guidance, the file-input path is unaffected.

Definition of done

Each is checkable pass/fail by running extract_spritesheet.py against the committed fixture and asserting on exported files + the manifest.

  • Keyed→transparent→assets: given a keyed source PNG with grid params, the entrypoint writes N individual PNGs whose four corner pixels are alpha=0.
  • Regular grid: on iso-4x4.png with --cols 4 --rows 4, exactly 16 assets are written and the manifest lists 16 elements with row/col each in 0–3, row-major.
  • Non-destructive: a run leaves the source file byte-identical and refuses to overwrite existing outputs unless --force is passed.
  • CLI entrypoint: a single documented extract_spritesheet.py runs end-to-end from the skill and --help lists every flag.
  • Manifest completeness: spritesheet.json carries per element — source sheet path, asset path, name, row/col, sheet_rect, trimmed_bounds, anchor/pivot, CSS background-position, and engine frame data — verified against a schema assertion.
  • Validation: --validate --expect 16 reports pass/fail for alpha corners, empty cells, key-color remnants, and asset count, exiting nonzero on any failure.
  • Keying modes: background removal works by supplied hex (--key #RRGGBB) and by border auto-sample (--key auto); --key none slices without keying.
  • Connected components: --slice components on a gridless sheet extracts each disconnected opaque blob as its own named element.
  • Naming: a provided --names list yields names like tile_01_grass.png; absent a list, stable auto names; ordering identical across repeated runs.
  • Export formats: --format webp and both produce alpha WebP alongside PNG.
  • Documented prompts: the skill documents prompts for generating easy-to-extract sheets (flat key bg, consistent grid, isolated non-touching elements).
  • Contact sheet: --contact-sheet preview.png writes a labeled QA preview image of the extracted elements.
  • Codex surface: agents/openai.yaml is present and well-formed, exposing the skill in the Codex CLI.
  • Generate-source composition: with --generate "<subject>", the entrypoint invokes codex-imagegen by name to produce a source sheet and then runs the full pipeline on it, recording source.generated: true and the subject in the manifest. Verified by a mocked/stubbed codex-imagegen invocation (the composition + handoff, not live pixel output); absent codex-imagegen, the flag fails with actionable guidance and the file-input path still works.

Test seams

  • Primary — the importable extract() API and the committed iso-4x4.png fixture: tests call extract(), then assert on the returned manifest dict and on the files written to a temp out dir (corner alpha via Pillow, element count, schema keys, row/col grid). Highest, most stable seam; no stdout parsing.
  • Secondary — the CLI: one end-to-end python3 extract_spritesheet.py … run plus --help, to cover argument wiring, non-destructive refusal, and exit codes (AC-3, AC-4, AC-6).
  • Deliberately untested: the aesthetic quality of despill fringing and contact-sheet layout — checked only structurally (no key remnants above threshold, file exists and is a valid image), not pixel-exact, since visual taste isn't mechanically gradable (per environment.md § Gaps).

Evidence required

  • Directory listing of the 16 exported PNGs (+ WebP for the format check).
  • The contact-sheet preview PNG.
  • The dual-executor probe transcripts + a per-criterion pass/fail table.
Risks
  • Auto border-sampling misfires on sheets whose border isn't the key color → mitigated by explicit --key #hex and a validation remnant check.
  • Connected-components merges touching iso tiles into one blob → grid mode is the default for regular sheets; components is opt-in for gridless input.
  • numpy 2.x / Pillow 12.x API drift on other machines → pin only a floor, keep to stable APIs, document deps.
Out of scope
  • Non-flat / gradient / photographic background removal — only a flat chroma key.
  • Animation assembly (GIF/APNG/spritesheet re-pack) — frame data is emitted, but no atlas repacking or animation output.
  • Normal maps, 3D, or multi-layer PSD input.
  • Implementing image generation itself — the optional --generate stage delegates to codex-imagegen by name; to-sprites owns no image model.
