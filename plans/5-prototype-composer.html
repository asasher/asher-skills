Extract prototype into a standalone thin-composer skill

The prototype technique is trapped inside the backlog monolith and its dev loop; it should be a standalone skill that answers one design question with a throwaway artifact — for a document, a maquette, or a UI, not only code.

issue #5enhancementplan v1

Problem & solution

Part of the asher-skills decomposition. backlog carries prototyping as an internal step: gates in reference/prototype.md, technique in the project playbook docs/agents/prototyping.md. It only ever fires from inside the enhancement branch, and its technique is written for code artifacts.

Solution: lift it into a self-contained skills/prototype/ skill — a thin composer that frames one design question, builds a throwaway artifact to answer it, presents it for feedback via review-loop, staffs the build via staffing (both by plain name), captures the answer, and deletes the artifact. Modeled file-for-file on the shipped skills/staffing/ and skills/review-loop/.

Composition model — the three-part dependency surface

Unlike the root primitives staffing and review-loop (sibling dependency: none), prototype is a composer: it declares real siblings and invokes them by name, importing no file.

User stories

  • As a developer planning an enhancement, I want to hand a blocking design question to prototype, so that a throwaway artifact settles it and its answer folds into my plan.
  • As a writer or designer outside a codebase, I want to prototype a document layout or design question, so that I can pick a direction from real variants without assuming any dev stack.
  • As a skill author, I want prototype to present its answer via review-loop and staff its build via staffing, so that I reuse the shipped primitives instead of forking review or routing machinery.
  • As a maintainer, I want the throwaway artifact deleted (or its validated core absorbed) once the answer is captured, so that nothing throwaway ships in the repo.
  • As a consumer installing the skill on a fresh repo, I want it self-contained, so that it works even with no docs/agents/prototyping.md playbook, degrading to sensible defaults.

Implementation decisions

  • Technique moves into the skill as a bundled reference. reference/prototyping.md becomes the authoritative, universal technique — this is what makes the skill usable outside dev. The project-playbook slot (docs/agents/prototyping.md) is declared but read only for repo-specific placement (task runner, routing, where throwaway artifacts live); absent one, the skill degrades to a self-contained artifact in a scratch/workspace dir rather than improvising repo structure.
  • Generalize beyond code. The opening reframes a prototype as a throwaway artifact answering one design question (medium usually but not only code). The two shapes generalize: a behavior/logic shape ("does this model/flow feel right under real cases?" — a driven reducer for code, a hand-driven state table or scenario run otherwise) and a form/look shape ("what should this look like / how structured?" — ?variant= on a real page for web UI, structural variant sheets for a document/layout).
  • Compose siblings by plain name, no imports. review-loop presents the answer (serve the artifact/answer sheet for annotated feedback; live interactive prototypes are driven directly but announced on the surface). staffing resolves who builds the artifact/variants (mechanical → gpt-5.5; ui/taste → Claude). Question-framing/diagnosis stays with the invoking thread. No path reference to any sibling file.
  • Preserve the four gates, generalized. Question stated → built & handed over → answer captured → cleaned up. Keep the "throwaway from day one — keep the answer, delete the artifact" framing verbatim in spirit.
  • Do not rewire or gut backlog (scope decision). backlog keeps its own reference/prototype.md and the full technique in docs/agents/prototyping.md until the separate backlog-dissolution work lands. The standalone skill carries its own bundled copy of the technique; the transient duplication is intentional so the monolith keeps working mid-decomposition. Rewiring backlog to compose the new skill is out of scope.
  • Ship the sibling artifacts. agents/openai.yaml and evals/probes.md mirror the shipped siblings so the skill is Codex-runnable and probe-gated before deployment.

Definition of done — acceptance criteria

This is a skills repo: "running the app" = file/structure checks + probe scenarios driven through an executor per environment.md § Driving the app. Each criterion is checkable pass/fail.

  • Layout. skills/prototype/ exists with SKILL.md, README.md, reference/prototyping.md, agents/openai.yaml, and evals/probes.md — matching the file layout of shipped skills/staffing/ and skills/review-loop/. Check: file listing.
  • Frontmatter. SKILL.md frontmatter parses and has name: prototype, user-invocable: true, an argument-hint, and a description stating it is a thin composer producing a throwaway artifact that answers one design question, usable beyond code. Check: YAML parse + content.
  • Three-part dependency surface. SKILL.md has a Dependency surface section declaring exactly three parts — bundled references, project playbook (docs/agents/prototyping.md), and sibling skills naming review-loop and staffing — and states prototype is a composer (siblings: not none). Check: content.
  • Composition by name only. No file in skills/prototype/ imports or path-references another skill's files (no skills/review-loop/… or skills/staffing/… path); the siblings appear by plain name. Check: grep across the skill dir.
  • Generalized beyond code. reference/prototyping.md opens by framing a prototype as a throwaway artifact (not only code) answering one design question, and gives at least one worked non-code example (e.g. a document/layout variant sheet). Check: content.
  • Framing & gates preserved. The skill keeps the "throwaway from day one — keep the answer, delete the artifact" framing and the four gates (question stated → built & handed over → answer captured → cleaned up). Check: content.
  • Probe (code question). Given a code design-question scenario, an executor (Claude subagent + gpt-5.5 via codex exec, dual-executor) produces a flow that states one question, picks the correct shape, presents via review-loop by name, staffs the build via staffing by name, captures the answer durably, and deletes the throwaway — graded pass against evals/probes.md. Check: probe transcript vs answer key.
  • Probe (non-code / outside dev). Given a non-code design question (e.g. which of three layouts a document should use), an executor produces a valid prototype flow with no codebase assumed — form shape, structural variants, present via review-loop — proving usability outside dev. Check: probe transcript vs answer key.
  • No regression to backlog. backlog's reference/prototype.md and docs/agents/prototyping.md are unchanged; backlog's prototype step still resolves its technique. Check: git diff shows those files untouched.

Test seams

Primary seam: the skill's files themselves (layout, frontmatter, dependency-surface content) checked directly, plus the evals/probes.md scenarios driven through the dual executor (Claude in-session + gpt-5.5 via codex exec) and graded against the answer key. Deliberately untested: the review-loop server and staffing resolver internals — they are shipped, separately probe-gated primitives; this skill only invokes them by name, so their behavior is out of scope here.

Evidence required

  • File listing of skills/prototype/ proving the layout (ac-1) and the grep proving name-only composition (ac-4).
  • Probe eval transcript + a pass/fail verdict table mapping each probe scenario to its answer-key criterion (ac-7, ac-8), from both executors.
  • Git diff confirming backlog's prototype files are untouched (ac-9).
Risks
  • Transient technique duplication (bundled ref vs backlog's playbook) drifts if edited in one place only — mitigated by the deferred backlog-dissolution work that removes backlog's copy.
  • Over-generalizing the technique could dilute the sharp two-shape guidance — mitigated by keeping the two shapes concrete with worked examples, code and non-code.
Out of scope
  • Rewiring backlog to compose the new skill, and slimming docs/agents/prototyping.md to repo-config-only — deferred to backlog-dissolution.
  • Changes to review-loop or staffing themselves.
  • The setup-asher-skills installer and any lockfile/registration wiring beyond the skill's own directory.
