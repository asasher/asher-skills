to-tickets — split a spec into backlog-ready tickets with blocking edges

A spec captures a decided direction but nothing pickup-able; someone still has to cut it into tracer-bullet tickets, order them, and wire the blocking edges by hand. to-tickets automates that split — the second half of the discussion → work bridge that to-spec (#8) opens.

issue #9enhancementplan v1

Where it sits

The consumer half of the spec→tickets pair. Its primary input is to-spec's output (a spec at docs/specs/<name>.md); it can also take a plan (#4) or the raw current conversation. It publishes into the bound tracker in dependency order, blockers first.

The split — vertical slices, with a wide-refactor exception

Default: draft tracer-bullet tickets — each a narrow-but-complete path through every layer, demoable on its own, sized to one fresh context window. The one exception is a mechanical, high-blast-radius change, which is sequenced instead of sliced.

Dependency edges + publish order

After the quiz settles granularity and edges, tickets are published blockers first so each blocker already has a tracker id its dependents can reference. Each dependent carries a - [ ] Depends on #N line — backlog's recorded convention (backlog-policy.md § Dependencies), which backlog run reads to skip blocked work.

Skill layout — mirrors the shipped primitives

Laid out to match skills/to-spec/ and skills/staffing/. Self-contained at the file level; composes siblings by name only.

User stories

  • As an agent holding a decided spec, I want to split it into tracer-bullet tickets with blocking edges, so that backlog can pick the work up in the right order without me wiring it by hand.
  • As the person driving the session, I want to be quizzed on granularity and blocking edges and iterate until I approve, so that the split matches my judgement before anything is published.
  • As a maintainer facing a wide mechanical change, I want it sequenced expand → migrate → contract instead of forced into a vertical slice, so that a high-blast-radius refactor lands safely in batches.
  • As backlog run, I want each dependent ticket to carry a Depends on #N edge in the recorded convention, so that I skip blocked work and dispatch blockers first.
  • As whoever grooms the backlog, I want readiness left to backlog groom by default, so that a fresh split doesn't auto-release work I haven't blessed — while still letting me apply it on approval when I want.
  • As the author of the source spec, I want to-tickets to never touch my spec or the parent issue, so that the source of truth is preserved.
  • As someone without a spec, I want to feed a plan or just the current conversation instead, so that I can still get a clean split.
  • As a Codex user, I want the skill to present correctly when installed into Codex, so that it works across harnesses.

Implementation decisions

  • New standalone skill skills/to-tickets/, laid out to match the shipped skills/to-spec/ and skills/staffing/ primitives (SKILL.md, README.md, agents/openai.yaml, reference/, templates/, evals/). Adapted from Matt Pocock's to-tickets and shipped as our own — the standing invariant: we never install an external skill. Self-contained at the file level; composes siblings by name only.
  • Primary input = a spec at docs/specs/<name>.md (to-spec's output). Also accepts a plan (#4) or the raw current conversation. The three inputs are read the same way — mine the decided direction, then split.
  • Draft vertical slices. Default output is tracer-bullet tickets: each a narrow-but-complete path through every layer, demoable on its own, sized to one fresh context window. The slicing reference makes this the default shape.
  • Wide-refactor exception. A mechanical, high-blast-radius change is sequenced expand → migrate-in-batches → contract instead of being forced into a vertical slice. The reference names the trigger and the three-phase shape.
  • Quiz the user — the human-confirmation step. Unlike to-spec (pure synthesis, no interview), to-tickets does interview: it quizzes the user on granularity and blocking edges and iterates until approved. Nothing publishes before approval.
  • Publish in the bound tracker's format, in dependency order. Publishing goes through the tracker binding recorded in docs/agents/platform.md (here: GitHub via gh). Tickets are created blockers-first so each Depends on #N resolves to a real earlier id.
  • Edges in backlog's recorded convention. Each dependent ticket carries a - [ ] Depends on #N body line per backlog-policy.md § Dependencies, so backlog run reads it and skips blocked work. The convention comes from a project playbook, not a skill import — to-tickets stays self-contained.
  • Generic "ticket" vocabulary. "Ticket" throughout; ticket == backlog's "issue" role (same thing, tracker-agnostic word). The skill never assumes GitHub-specific vocabulary in its own text.
  • Readiness default = leave it to backlog groom. A fresh split does not auto-apply ready-for-agent; the skill notes the option to apply it on approval (Matt's posture) but recommends leaving it to groom. This is the decided default.
  • No stale content; never modify the parent. Tickets carry no file paths or code snippets (the single prototype-validated-snippet exception is preserved). The skill never edits the source spec or the parent issue.
  • Three-part dependency surface declared in SKILL.md: bundled references (ticket + tickets-file templates, the slicing/template guides), project playbooks (backlog-policy.md § Dependencies convention, platform.md tracker binding, the presentation/vocabulary conventions), sibling skills (consumes to-spec; can also take a plan (#4) or a raw conversation — inputs composed by name, no hard skill import).
  • Codex manifest agents/openai.yaml per docs/patterns/codex-compat.md, allow_implicit_invocation: false (an operator skill that publishes to a tracker, not implicitly fired).
  • Pre-deployment probe eval evals/probes.md: situated dry-run probes with a pre-written answer key, dual-executor (Opus in-session + codex exec), per docs/patterns/probe-evals.md, covering the criteria below.

Definition of done — acceptance criteria

Each is checkable pass/fail: situated probe dry-runs against both executors (Opus in-session + codex exec) with correct-action-and-citation grading, plus file/grep checks for the structural ones.

  • Skill shape. skills/to-tickets/SKILL.md frontmatter identifies a user-invocable skill (name: to-tickets, user-invocable: true, a description that reads as splitting a spec/plan/conversation into backlog-ready tickets); the directory mirrors the shipped primitives (SKILL.md, README.md, agents/openai.yaml, reference/, templates/, evals/); and no file in the skill imports or reads another skill's files (grep finds no cross-skill paths).
  • Dependency surface. SKILL.md declares all three pointer kinds — bundled references, project playbooks (naming backlog-policy.md § Dependencies and the platform.md tracker binding), and sibling skills — and states the sibling relationship as consumes to-spec's output; also accepts a plan (#4) or a raw conversation, composed by name, self-contained at the file level.
  • Inputs. A probe confirms the primary input is a spec at docs/specs/<name>.md, and that a plan or the current conversation are accepted alternates. Citable.
  • Vertical slices. A probe on a multi-layer feature resolves to: draft tracer-bullet tickets — each a narrow-but-complete path through every layer, demoable on its own, sized to one fresh context window. Citable.
  • Wide-refactor exception. A probe on a mechanical, high-blast-radius change resolves to: sequence expand → migrate-in-batches → contract instead of forcing a vertical slice. Citable, with the trigger named.
  • Quiz the user. A probe confirms to-tickets quizzes the user on granularity and blocking edges and iterates until approved — the human-confirmation step — and that nothing publishes before approval. It explicitly does not behave like to-spec's no-interview rule. Citable.
  • Dependency edges, correct convention + order. A probe splitting a spec into ≥3 tickets where two depend on a third emits each dependent's edge as a - [ ] Depends on #N body line per backlog-policy.md § Dependencies, publishes blockers first so the ids resolve, and states the edges let backlog run skip blocked work. Correct edges + order = pass; wrong convention, wrong direction, or blocker-last = fail.
  • Publish through the bound tracker. A probe confirms publishing goes through the tracker binding in docs/agents/platform.md (GitHub via gh here), in dependency order, using generic "ticket" vocabulary (ticket == backlog's issue role).
  • Readiness default. A probe confirms the default is to leave ready-for-agent to backlog groom, with the noted option to apply it on approval. Choosing to auto-release by default = fail.
  • No stale content; never modify the parent. A probe confirms tickets carry no file paths / code snippets (prototype-validated-snippet exception preserved) and that the skill never edits the source spec or parent issue. Citable.
  • Codex manifest. skills/to-tickets/agents/openai.yaml parses and is well-formed per docs/patterns/codex-compat.md, with allow_implicit_invocation: false.
  • Probe eval ships. skills/to-tickets/evals/probes.md exists with a pre-written answer key covering ac-1..ac-11, dual-executor (Opus in-session + codex exec), per docs/patterns/probe-evals.md.

Test seams

This ships a skill (a prompt-driven procedure), so the seam is the probe eval, not a code test — the highest existing seam for skills in this repo (environment.md § Driving the app, docs/patterns/probe-evals.md). Every acceptance criterion is exercised by at least one situated dry-run probe against both executors, graded correct-action-and-citation; structural criteria (ac-1, ac-2, ac-11, ac-12) are additionally confirmed by file / grep / YAML-parse check. The two behaviour-heavy probes — the spec→tickets split with correct Depends on #N edges (ac-7) and the wide-refactor expand/migrate/contract path (ac-5) — are the load-bearing ones. No new script or runtime is introduced, so there is nothing else to seam.

Evidence required

  • The probe-eval transcript from both executors (Opus in-session + codex exec) with a pass/fail verdict table mapping each probe → its criterion.
  • A grep / YAML-parse check log for the structural criteria (no cross-skill imports, three-part dependency surface present, openai.yaml well-formed).
  • A sample split produced from a real spec (this decomposition spec is the canonical input) showing ordered tickets with correct Depends on #N edges, plus one wide-refactor case showing the expand/migrate/contract sequencing — rendered end-to-end without publishing to the live tracker.
Risks
  • Publishing writes to the live GitHub tracker (real issues). The probe eval is read-only / dry-run (executors describe the next action + cite), so no test run creates stray issues; the sample split (ev-3) renders to the tickets file, not the tracker.
  • The quiz (interview) is the opposite posture to to-spec (no interview). Mitigation: the SKILL.md and slicing reference state the contrast explicitly, and ac-6 grades it.
  • "Vertical slice" vs "wide-refactor exception" can blur on a change that is partly both. Mitigation: the reference names a concrete trigger for the exception (mechanical + high blast radius) and ac-5/ac-4 grade both branches.
  • Blocker-first ordering depends on the tracker assigning ids at creation. Mitigation: the method makes ordering a publish-time step, and ac-7 grades that blockers are created first.
Out of scope
  • Changing backlog's dependency convention or the run selection logic — to-tickets emits into the existing convention, it doesn't redefine it.
  • Standardizing the whole repo on "ticket" instead of "issue" — the issue notes this as eventual, not required here.
  • Grooming / releasing the tickets (ready-for-agent) — left to backlog groom by default (dec-9).
  • Installing or vendoring Matt Pocock's skill — we adapt and ship our own; no external skill is installed.
  • Changes to to-spec, plan, or the backlog loop themselves.
