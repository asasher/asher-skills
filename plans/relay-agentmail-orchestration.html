Relay — direct, governed project communications

People need dependable project updates without the current two-provider forwarding detour, while the person running the workflow retains final control over exactly what is sent and to whom.

plan v3name: Relayimplementation directed

The shape

Relay turns verified evidence into an immutable, human-approved email unit; AgentMail then becomes the sole supported delivery and lifecycle provider.

Stories

  • As a project operator, I want Relay to gather only attributable updates and assemble the right sections for each audience, so that recipients get relevant, trustworthy communication.
  • As the person running Relay, I want to inspect and explicitly approve the exact sender, To, CC, and rendered email before it sends, so that the agent never surprises me or contacts the wrong people.
  • As a project maintainer, I want setup to bind Relay to this repository’s projects, audiences, evidence sources, and editorial rules, so that the portable skill does not contain private or project-specific facts.
  • As an operator, I want sent, delivery-failure, and reply state correlated to the original communication, so that follow-up work is visible without duplicate sends.

Key decisions

  • The skill and its instance are Relay. Rename the source, installed package, interface, project playbook, and consumer instance root together. The portable skill starts and resumes only from control-plane/relay/; previous communications state is reconciled only when a project owner explicitly requests that one-time rollout.
  • One provider, draft-first delivery. AgentMail is the only supported provider. Create the exact approved message as a draft with a deterministic client ID derived from the communication and approved hashes, verify the returned draft fields, record its draft ID, then send that draft. Every retry reuses the same IDs; an ambiguous result blocks for reconciliation rather than minting a second send identity.
  • Review-loop is required and the review sheet is self-contained. Relay embeds every email's actual HTML/text, forced light/dark previews, sender, To, CC, and canonical manifest hashes into one review document. `review-loop` owns serving, annotations, the hash-bound verdict, and awaiting. Before provider work, Relay recomputes the run hashes and requires an exact match to the approved embedded manifest.
  • The proven template structure ships; identity stays local. Relay ships an unbranded default extracted from the current workspace’s compact project-section template, including authored light/dark modes and a plain-text twin. The default remains a consumer-local, version-pinned React Email renderer; setup verifies its Node runtime and materializes it only when missing. Local bindings own identity, assets, footer copy, and extensions, and setup never overwrites them.
  • AgentMail identity is project-scoped. Setup reads or provisions only AGENTMAIL_API_KEY in the consumer repository’s root .env, so each repository deliberately selects its AgentMail account/inbox. The key is never committed, printed, copied into state, or passed on argv; setup checks that .env is ignored and owner-readable only. Non-secret sender, inbox, and capability metadata remain in control-plane/relay/.
  • Evidence sources are project bindings. Relay itself requires `review-loop`; setup records the authoritative paths, commands, connectors, or sibling skills for each evidence class. Existing `manage-tasks` and `manage-opportunities` integrations remain usable when bound but are not universal hard dependencies.
  • Bindings are data, not prompt lore. The consumer-owned instance contains projects, audience membership, per-recipient header roles, interest/disclosure rules, section recipes, sender identity, default operator CC, and update watermarks. Each run snapshots and hashes the resolved manifest.
  • Operator CC is mandatory by default. Setup identifies the operator address and appends it to CC for every external send unless the human explicitly changes that policy for a named audience. Existing internal/external audience labels do not imply To or CC: recipient header role is explicit. The sender is not automatically a recipient.
  • The local ledger separates workflow, recipient delivery, and conversation facts. Workflow progresses through selected, rendered, reviewed, draft-created, send-submitted, and sent or an explicit blocked/rejected/superseded state. Delivery/bounce/complaint facts are recorded per recipient; replies are thread events. A message is never called fully delivered unless every intended recipient has a receiving-server delivery event.
  • Selection watermarks advance on confirmed send, not delivery. Advance only after `message.sent` or unique sent-message reconciliation for the approved provider message. Draft creation and send submission do not advance it; later bounce or complaint becomes visible follow-up work and never triggers an automatic resend.
  • Custom domain is an optional setup branch. Setup offers it, records verification status, and allows a user to defer it. Live send uses only the configured verified sender identity.
  • Event intake is capability-tiered. Setup prefers a signature-verified AgentMail webhook receiver and otherwise records manual/on-demand reconciliation. “Delivered” means accepted by the receiving mail server, never opened or read. Relay never claims real-time tracking without an effect-verified receiver.

Definition of done

  • The published package and interface are named Relay, require `review-loop`, contain no Outlook delivery path, and create or resume project material only under control-plane/relay/ with docs/agents/relay.md.
  • On each of two temporary repositories with different project/evidence layouts, `setup` discovers available sources and produces a complete local binding for projects, evidence providers, section recipes, audiences, To/CC roles, operator CC, cadence, sender, template, and reconciliation mode.
  • Setup reads or provisions only `AGENTMAIL_API_KEY` in a repository-root, gitignored, mode-0600 `.env`, preserves unrelated assignments, and completes capability validation without exposing or persisting the key in argv, output, logs, artifacts, or state.
  • Given two external audiences and one internal audience with different project, interest, disclosure, section, and header-role rules, selection produces exactly one validated bag or explicit exclusion per eligible audience with no cross-audience fact or recipient leakage.
  • A fresh instance renders one project update and one internal digest with the unbranded compact default, semantically equivalent HTML/text, and readable forced light/dark previews; the manifest records the pinned local renderer/template identity.
  • The self-contained review sheet visibly includes every message’s actual HTML/text, sender, To, CC, evidence summary, forced light/dark previews, and canonical hashes that the send adapter independently recomputes.
  • For an approved controlled-inbox message, Relay creates one AgentMail draft using a deterministic client ID, verifies the draft fields, records its draft ID before send, and sends that exact draft with the policy-required operator CC.
  • Across duplicate inputs and injected timeouts before draft creation, after draft creation, and after send submission, every retry reuses the same client/draft identity and the controlled inbox observes at most one sent message; an unreconciled outcome becomes blocked rather than recreated.
  • Duplicate and out-of-order sent/delivered/bounced/complained/rejected/reply events are idempotently correlated by AgentMail identifiers, while a mixed-recipient fixture retains distinct delivery outcomes and never reports all-delivered prematurely; without an event receiver, the same facts reconcile on demand and are never described as real time.
  • Setup offers optional custom-domain verification and blocks live sending from an unverified selected identity, while a user who declines a custom domain can complete setup with another verified AgentMail sender.
  • The deterministic suite, controlled-inbox loopback, and prewritten dual-executor probe evaluation pass, including secret failure, audience isolation, stale review, retry ambiguity, mixed delivery, and absent-event-receiver scenarios.
  • An audience watermark advances only after a matching AgentMail `message.sent` event or unique sent-message reconciliation; draft creation, submission, delivery, bounce, complaint, and reply events do not independently advance it or trigger automatic resend/reply.
  • A scheduled run with new evidence may select, render, and present review, but performs zero provider writes until the exact current review is approved.
  • Rerunning `setup` after package defaults change preserves every deliberate local value and modified template, while reporting candidates or blocking conflicts instead of overwriting them.
  • Changing rendered HTML, text, sender, To, CC, or the embedded manifest after review invalidates delivery authorization and causes zero AgentMail writes until a new hash-bound `review-loop` approval exists.

How it will be checked

  • Temporary-repository setup tests cover a fresh generic project and a scrubbed copy of the current Asher workspace binding, then rerun setup after local edits and changed defaults.
  • Renderer fixtures compare semantic HTML/text content, inspect forced light/dark screenshots, and verify the unbranded default contains no Dunn Harland assets or copy.
  • A review-loop fixture mutates each approved input—HTML, text, sender, To, CC, and manifest—and asserts the provider shim remains untouched until fresh approval.
  • A fake AgentMail boundary injects failures at every draft/send seam; a controlled dedicated inbox confirms the final at-most-once behavior and provider correlation without stakeholder addresses.
  • Signed webhook fixtures exercise duplicate, out-of-order, mixed-recipient, bounce, complaint, rejection, and reply events; the same cases run in manual reconciliation mode.
  • Prewritten answer keys grade two isolated executor runs for disclosure, scheduling, idempotency, credential safety, approval safety, and truthful lifecycle reporting.

Open/click analytics, automated reply generation, and hosting a production webhook receiver are deliberately not checked in this first change.

What's required

  • A controlled AgentMail account with a dedicated inbox, scoped test key, verified sender, and controlled recipient addresses for loopback—never real stakeholder recipients.
  • The required `review-loop` sibling and its configured presentation surface.
  • Node/npm compatible with the pinned consumer-local React Email renderer; setup must verify this before promising render capability.
  • A prewritten probe answer key and immutable fixtures derived from both a generic repository and the current communications instance.

Risks and mitigations
  • Direct send raises blast radius. Mitigate with a self-contained hash-bound review sheet, least-privilege credentials, deterministic drafts, a controlled loopback, and blocked-safe ambiguity handling.
  • A top-level review hash can miss linked preview-file changes. Embed the reviewed content and canonical manifest in the review sheet, then recompute run hashes before provider work.
  • Webhook hosting is not inherent to a portable skill. Make it a consumer capability with manual reconciliation as a truthful fallback; never create a public tunnel automatically.
  • A one-time rollout can leave two writable histories. New Relay code reads and writes only control-plane/relay/; any prior-version reconciliation requires an explicit project instruction, a validated cutover, and deliberate handling of the untouched old directory outside the portable skill.
  • Evidence selection may leak irrelevant project details. Bind authoritative sources during setup, preserve audience interest/disclosure policy, expose the evidence summary in review, and test audience isolation.
  • A message may partly deliver. Track recipient outcomes separately and reserve all-delivered for the complete intended set.
Out of scope
  • Alternative email providers or a pluggable provider abstraction.
  • Outlook forward drafts, Sent Items reconciliation, or any Outlook dependency.
  • Automated replies, autoresponders, open/click analytics, CRM synchronization, and bulk-campaign features.
  • Hosting a production webhook receiver for every consumer; Relay specifies the capability contract and verifies the consumer-selected endpoint when present.
  • BCC, attachments, and scheduled future delivery in the first Relay release; each expands the review and lifecycle contract and requires a separate approved plan.
