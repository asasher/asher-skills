Extract staffing into a standalone, global-capable primitive skill

The model-roster machinery — roles, rankings, routing, fallback — is trapped inside the backlog dev monolith, so non-dev projects and other people can't borrow it, and its table is hardcoded to one machine.

issue #1enhancementplan v1decomposition · move #1

What we're building

A new standalone skill skills/staffing/ that owns the staffing primitive: it derives its roster from a fresh machine audit (not Asher's baked-in models), installs a global base with sparse project overrides, keeps capabilities and task-pins as first-class structures separate from the higher=better rankings, and reconciles by LLM audit with no version stamps. This is the cheap proof-of-shape for the whole backlog decomposition. Rewiring backlog to consume it by name is issue #6, out of scope here.

Two-layer install

A machine audit fills a global base (harness-coupled) once; each project carries only its deltas.

How a staffing question resolves

Pins and capabilities are checked before the ranking tie-break — they are gates, not rows in the rankings table.

User stories

  • As an agent mid-workflow, I want to resolve "who should do this task" against one roster, so that delegation is consistent across the loop and ad-hoc sessions.
  • As someone installing from the public repo on a different machine, I want the skill to audit my reachable models and Codex install, so that the roster fits my environment instead of Asher's.
  • As a user with existing global staffing rules, I want the setup flow to show them and offer project overrides, so that I extend rather than clobber.
  • As a user with a project that differs only slightly (e.g. a stricter floor), I want to record just the delta, so that the override file stays small and legible.
  • As an agent routing a browser-driving or computer-use task, I want capabilities checked as booleans separate from the rankings, so that the intelligence > taste > cost tie-break stays intact.
  • As a user re-invoking the skill later, I want it to audit installed-vs-repo and report drift or conflict, so that I trust the config without hunting for version markers.
  • As a Codex-driven session, I want the skill to present correctly, so that Codex threads read the same roster.
  • As a composer skill (plan, prototype, backlog), I want to reference staffing by plain name and find its dependency surface declared, so that I can rely on it without importing files.

Implementation decisions

  • Source home. The skill's canonical source is skills/staffing/ (repo convention; .claude/skills/ and .agents/skills/ are install mirrors). It ships SKILL.md, README.md, bundled reference/, agents/openai.yaml, and evals/.
  • Content lift, then generalize. The roles + fallback ladder (from backlog/reference/staffing.md), the rankings + routing rules (picking-models.md), and the harness mechanics (claude-md-overlay.md) move into bundled references. Asher's concrete four-model table becomes an example of audit output, never the authoritative shipped table.
  • Machine-audit procedure. A documented, prompt-driven procedure the skill runs at install: probe which models the harness can reach and whether Codex is installed, then write the rankings table from that. No scripts required — it is agent-driven, matching the LLM-audit posture. Cost/taste/intelligence numbers for audited models come from a documented default seed the user can tune, since those are judgments the machine can't probe.
  • Two layers, delta-only overrides. Global base lives harness-coupled: ~/.claude/CLAUDE.md plus a global AGENTS layer, holding the audited table, generic routing, pins, and CLI/tools mechanics. Project overrides live in the project's docs/agents/ (and repo memory files) and carry only deltas — a resolver reads base, then applies overrides.
  • Capabilities as a separate matrix. Computer-use / browser-use are boolean columns in their own matrix, keyed by model, never merged into the higher=better rankings table. Routing checks the matrix as a filter gate (diagram 2, step 2).
  • Task-type pins as a first-class list. Pins (e.g. mechanical/bulk → gpt-5.5) are an explicit named list resolved before ranking (diagram 2, step 1), distinct from the ranking-derived defaults.
  • Scope-decision flow. On setup/invoke: if global rules already exist, show them and offer to add project overrides; if none exist, ask global-with-overrides vs project-only. Branch is chosen from the audit of what's already installed.
  • Reconciliation is LLM audit. Re-invoking compares installed base + overrides against the skill's repo definition and reports drift/conflict in prose. No version stamps or markers are introduced as the reconciliation mechanism (a deliberate departure from backlog's vNN template stamps).
  • Dependency surface declared. SKILL.md names all three pointer kinds: bundled references (own contract), project playbooks it installs into docs/agents/, and sibling skills (none required — staffing is a root primitive). Self-contained at the file level; composes by name.
  • Codex compatibility. Ships agents/openai.yaml per docs/patterns/codex-compat.md, because Codex-driven threads must read the same roster.
  • Pre-deployment probe eval. An evals/ scenario set + answer key per docs/patterns/probe-evals.md, run dual-executor (Opus in-session + gpt-5.5 via codex exec) before first real use. This is the verify/evidence contract for the criteria below.

Definition of done

Each is checkable pass/fail by a probe-eval executor reading skills/staffing/ and/or by a structural file check (per environment.md § Driving the app).

  • Skill exists and is self-contained. skills/staffing/SKILL.md exists with valid frontmatter (name: staffing, invocable, global-capable description); the directory imports no other skill's files (grep finds no cross-skill file paths).
  • Dependency surface declared. SKILL.md explicitly names all three pointer kinds — bundled references, project playbooks (under docs/agents/), sibling skills (stating "none — root primitive").
  • Roles + fallback preserved. A probe posing a staffing question the old backlog/reference/staffing.md answered (e.g. "the ui builder is unreachable — who takes it?") gets an equivalent answer from the new skill, citing the succession ladder.
  • Rankings vs capabilities are separate structures. The rankings table contains only cost/intelligence/taste; computer-use and browser-use appear only in a distinct capability matrix. A probe routing a browser-driving task resolves via the matrix as a gate, then the intelligence > taste > cost tie-break, and the executor points to each structure by location.
  • Task-pins distinct from ranking routing. A named pin list exists; a probe ("who does a bulk mechanical edit?") returns the pinned model and the executor cites the pin, not a ranking derivation.
  • Machine-audit procedure, not a hardcoded roster. The skill body carries an audit procedure that probes reachable models + Codex presence and writes the table from it; no fixed machine-specific roster is shipped as authoritative (Asher's models appear only as labeled example output). A dry-run probe on a hypothetical different machine (no Codex, different models) follows the procedure and produces a coherent table.
  • Two-layer install with delta-only overrides. The skill specifies a global base (~/.claude/CLAUDE.md + global AGENTS layer) and project overrides carrying only deltas. A probe ("install for a project whose only difference is a stricter floor") produces an override containing just that delta, not a full copy of the base.
  • Scope-decision flow both branches. A probe where global rules already exist drives the "show + offer overrides" branch; a probe with none drives the "ask global-with-overrides vs project-only" branch. The executor takes the correct branch in each.
  • LLM-audit reconciliation, no version stamps. Re-invoking against a deliberately-drifted installed section detects and reports the drift/conflict in prose; the skill introduces no vNN-style stamp as its reconciliation mechanism (grep confirms none).
  • Codex compatibility. skills/staffing/agents/openai.yaml exists and is well-formed per docs/patterns/codex-compat.md.
  • Probe eval green. skills/staffing/evals/ ships a scenario set + answer key; the dual-executor run (Opus in-session + gpt-5.5 via codex exec) passes every probe, with a pass/fail verdict table mapping each probe to its criterion.

Test seams

The skill is a prompt-driven procedure, so the criteria are tested at two seams: probe scenarios in evals/ driven through an executor model and graded against the answer key (ac-3 through ac-9), and structural file checks — presence, frontmatter, grep for cross-skill imports / version stamps, YAML well-formedness (ac-1, ac-2, ac-6's no-hardcode clause, ac-10). No runtime app, no scripts to unit-test (reconciliation is deliberately LLM-driven, not scripted). Deliberately untested: the subjective quality of the seeded cost/taste/intelligence numbers — those are user-tunable judgments, not mechanically gradable.

Evidence required

  • The dual-executor eval transcript (Opus in-session + gpt-5.5 via codex exec).
  • A pass/fail verdict table mapping each acceptance criterion to its probe or file check.
  • The rendered plan screenshot (for this plan's own approval record).

Risks
  • Global install has real side effects on ~/.claude/; mitigated by auditing before writing and by Q2's scratch-path eval.
  • LLM-audit reconciliation is fuzzier than version stamps; mitigated by ac-9's deliberately-drifted probe proving detection works.
  • Over-generalization — building for hypothetical machines we can't test. Mitigated by the ac-6 dry-run probe standing in for a second environment.
