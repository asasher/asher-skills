setup-asher-skills — the prompt-driven installer

Wiring the right asher-skills into a new project today means hand-copying skills, guessing which siblings each one needs, and hand-writing the context blocks and playbooks — error-prone, and nobody remembers the dependency graph.

Solution: one prompt-driven skill that audits the repo, the machine, and the project, interviews the user one plain-language decision at a time, then installs a dependency-complete skill set from this repo only and writes the project's skill map — and on re-invocation, audits the install for drift.

#7plan v1enhancementcapstone

The shape — four phases, stop at confirm

Explore three surfaces silently, then interview one decision at a time, confirm the whole write, and only then touch disk. Modeled on setup-matt-pocock-skills (explore → decide-one-at-a-time → confirm → write), adapted as ours.

The dependency graph it guarantees

Accepting a composer silently pulls its siblings — the invariant the skill enforces so no half-wired install ships. Two root primitives depend on nothing; everything else closes over them.

Stories

  • As a developer starting a fresh project, I want the installer to audit my repo, my machine, and ask what the project is for, so its recommendation fits before it writes anything.
  • As a user who doesn't know the skill names, I want each decision presented one at a time with a plain-language explainer, so I can say yes or no without reading docs.
  • As someone who accepts plan, I want review-loop and staffing pulled in automatically and told to me, so I never end up with a half-wired composer.
  • As the repo owner, I want every install to come from this repo's public endpoint only, so no external skill is ever silently added.
  • As a returning user, I want the installer to prefer project-local scope and only offer global for a genuinely reusable skill (staffing) with my consent, so my home directory stays clean.
  • As a user re-running months later, I want an audit of what drifted from the live repo — not a version-number check — with fixes proposed one at a time.
  • As the next agent working this repo, I want a ## Agent skills map and a repo pointer written into the harness memory file, so I know what's installed and where it came from.

Key decisions

  • Compose, don't reimplement, the machine audit. The machine half of the audit invokes the staffing skill by name for the reachable-models + Codex probe, rather than re-deriving a roster — staffing owns that, including the consent-gated global write.
  • setup authors only the cross-cutting artifacts; skills own their own playbooks. setup writes the ## Agent skills map and the repo pointer directly (no skill owns these), and guarantees the docs/agents/ playbooks by running each installed skill's own setup (staffing roster, review-loop surface config, backlog's docs/agents suite) — it never re-copies another skill's templates. This keeps composition by-name and avoids template drift.
  • Project-first scope. Everything installs project-local by default. Global is offered only for staffing (the one genuinely cross-project roster) and only with consent, delegated to staffing's own scope-decision flow. No other skill is offered global.
  • Guarantee the closure at accept time. Accepting a composer computes its transitive sibling closure and installs it: plan|prototype ⇒ ensure review-loop + staffing; backlog ⇒ ensure {staffing, review-loop, plan, prototype}; maquette ⇒ ensure review-loop. The pull is surfaced in plain language, never silent-and-hidden.
  • This-repo-only is a hard invariant. Every install command targets https://github.com/asasher/asher-skills and nothing else. External good ideas are already adapted and shipped here (e.g. to-spec), so the skill never emits an install for mattpocock/* or any other host — it offers our equivalent or says we don't ship it.
  • Reconciliation is an LLM audit, no version stamps. Re-invoke fetches the repo's current skill catalog, reads what's installed (project skills-lock.json + .claude/skills/, global ~/.claude/skills/), reads the ## Agent skills block, and reports overlap/drift/broken-closure/scope-conflict in prose. No vNN stamp, no hash marker as the drift mechanism — the model reads and judges (matching staffing/review-loop, rejecting backlog's stamp idiom).
  • The context block is a map, not a router. The ## Agent skills section is a plain per-project list (skill + one-line purpose + the sibling relationships + the repo pointer), written into whichever memory file exists (prefer AGENTS.md, else CLAUDE.md). No separate ask-asher dispatcher skill.

Definition of done — acceptance criteria

Two seams: structure/grep checks against the skill's files, and dual-executor Tier-1 probes (Opus in-session + gpt-5.5 via codex exec), each graded pass/fail against an answer key written before any run. A probe passes only if both executors pass it.

Structure & invariant checks (mechanical)

  • fileskills/setup-asher-skills/SKILL.md exists with valid frontmatter: name: setup-asher-skills, a description, an argument-hint, and user-invocable: true.
  • fileSKILL.md has a ## Dependency surface section declaring all three parts: bundled references (interview + catalog + audit-mode + the seed templates), the project artifacts it writes (the ## Agent skills block + the guaranteed docs/agents/ playbooks + the pointer), and the sibling skills it installs/guarantees.
  • fileThe layout exists: reference/interview.md, reference/catalog.md, reference/audit-mode.md, templates/agent-skills-block.md, templates/repo-pointer.md, evals/probes.md, and agents/openai.yaml are all present.
  • grepEvery install/pull command across the skill's files references only asasher/asher-skills (slug or full github.com/asasher/asher-skills URL); grep finds no other host or repo slug inside an install command.
  • grepThe sibling-closure rules are stated in reference/catalog.md: text asserting plan|prototype ⇒ review-loop + staffing, and backlog ⇒ {staffing, review-loop, plan, prototype}.
  • grepreference/audit-mode.md contains no version-stamp mechanism — no instruction to write or compare a vNN / -templates: / hash marker as the drift signal — and states reconciliation is an LLM audit.
  • fileevals/probes.md defines at least the six probes below, names both executor targets (Opus in-session + codex exec), and keeps its answer key separate (not inline with the probes).

Behavioral probes (dual-executor; both must pass each)

  • probe P1Invoked fresh, the executor audits three surfaces — repo, machine (by composing staffing for the model/Codex probe), and user/project — and cites the file+sentence that says so.
  • probe P2Given a user who accepts plan, the executor installs plan and auto-pulls review-loop + staffing, presents it one decision at a time with a plain-language explainer, and tells the user the siblings were pulled.
  • probe P3Asked for an external skill (e.g. Matt Pocock's TDD), the executor refuses to install from any host but asasher/asher-skills — offering our adapted equivalent or stating we don't ship it — and never emits an external install command.
  • probe P4After the interview, the executor writes a ## Agent skills block into AGENTS.md/CLAUDE.md (prefer existing), guarantees the docs/agents/ playbooks via each installed skill's setup, and writes the repo pointer — with no ask-asher router.
  • probe P5Re-invoked on a repo it set up before, the executor runs audit mode: fetches the repo's current catalog, diffs against installed (project + global), reports overlap/drift/broken-closure/scope-conflict, and proposes updates one at a time — using no version stamps.
  • probe P6Given a user accepting staffing and backlog, the executor scopes everything project-local by default and offers staffing global only with consent, keeping backlog and the rest project-local.

How it will be checked

AC-1…7 are confirmed at the file seam — ls/grep/read against skills/setup-asher-skills/, no model in the loop. AC-8…13 are confirmed at the probe seam per docs/patterns/probe-evals.md: each probe is run in a Claude subagent (Agent tool, in-session) and via codex exec -s read-only, graded against the pre-written key; both executors must pass. Deliberately unchecked: the subjective quality of the interview copy and the recommendation's taste — those get a human review pass on the presentation surface, not a mechanical grade (consistent with environment.md § Driving the app, "a skill whose value is subjective can't be fully graded mechanically").

What's required on completion

  • The eval transcript (both executors) plus a per-probe pass/fail table mapping each probe to its answer-key criterion — the evidence evidence.md consumes.
  • A screenshot of the rendered plan is not required for the skill itself; the skill produces no visual artifact of its own (it writes markdown + runs installs).
Risks & mitigations
  • Composition drift: setup could accidentally re-copy a sibling's playbook templates and rot. Mitigation: dec-2 — setup delegates to each skill's own setup and authors only the map+pointer; AC checks it writes no sibling template verbatim.
  • Executors misread "audit three things" as one big audit. Mitigation: P1 requires citing the exact sentence; if executors conflate, tighten interview.md wording before ship (probe-evals loop).
  • Global write surprises the user. Mitigation: dec-3 routes any global write through staffing's consent gate; P6 checks it.
  • The npx skills tool's exact flags (-g/-y) or lockfile behavior differ from assumption. Mitigation: verify the command form against the repo README + a dry run before ship; the skill states the command, not a bespoke installer.
Out of scope
  • The meta dogfood — removing backlog from this repo's working checkout and running setup-asher-skills against this repo — is a separate follow-on step, not this issue's build.
  • Changing any installed skill (staffing, review-loop, plan, prototype, backlog, etc.). This issue ships the installer that composes them as they are.
  • A new npx skills installer binary or lockfile format. The skill drives the existing skills tooling; it does not replace it.
  • An ask-asher router skill. Explicitly excluded — the ## Agent skills block is the per-project map.
