Harden setup-asher-skills audit-mode: 5 dogfood gaps

Re-invoking the installer on a repo that already has asher-skills (including its own home) can never reach audit-mode, and even when it does the audit is blind to the Codex harness dir, cross-harness duplicates, foreign-source installs, and the self-catalog case.

#27enhancementplan v1

The sharpest fix — routing (gap A)

Route on evidence of a prior install, not on the ## Agent skills block alone; a missing block becomes an audit finding, not a one-way ticket back to setup.

Stories

  • As someone who set a repo up with the old backlog (skills but no map block), I want a re-invoke to run the audit, so that the installer can reconcile my drift instead of looping back to a fresh setup.
  • As a maintainer auditing a dual-harness repo, I want .agents/skills/ read alongside .claude/skills/, so that Codex-side copies aren't invisible to the audit.
  • As a maintainer, I want a skill installed in both harness dirs called out as its own overlap category, so that I know whether it's a benign symlink or two copies that can drift apart.
  • As a maintainer, I want a pre-existing external install (e.g. writing-great-skills from mattpocock/skills) surfaced by the audit, so that the pull-only invariant is enforced on reconcile, not just on install.
  • As someone auditing the skills repo itself, I want the local skills/ dir used as the catalog, so that a local branch ahead of origin doesn't produce false "new in repo" / drift findings.
  • As a maintainer, I want each fix confirmed by a probe that two different executor models both pass, so that the contract change is real behavior, not just reworded prose.

Key decisions

  • Route on install-evidence, three states. The block stops being the sole routing signal. The predicate is "any evidence of a prior asher-skills install." Forced by gap A: the block is written only at the end of a setup, so a legacy or self-host install has skills but no block and must still reach audit.
  • A missing block is an audit finding, not a setup trigger. Rather than a fourth "repair" mode, the populated-but-no-block case enters audit, and audit gains a Missing map finding whose proposed fix writes the block. Keeps two run-modes (setup/audit) and reuses audit's one-fix-at-a-time discipline.
  • Harness dirs are enumerated, not discovered. The contract names the exact set to scan — .claude/skills/ and .agents/skills/ at project, ~/.claude/skills/ and ~/.agents/skills/ at global — and resolves symlinks before comparing, so a symlinked pair is one skill with two mount points, not two installs. Chosen over globbing "any */skills/" to stay explicit and predictable.
  • Cross-harness duplication splits benign from divergent. Symlinked pair (one underlying file, this repo's own layout) → note only, not drift. Two independent copies that can diverge → flag with a recommendation to consolidate. Prevents the audit from crying wolf on the intended dual-harness symlink layout.
  • Foreign source is read from skills-lock.json. The lock file records each skill's source repo; the audit keys the finding off source ≠ asasher/asher-skills. Surface-and-advise, never auto-remove — an external skill may be intentional; the pull-only invariant governs what setup installs, and the finding offers our adapted equivalent (or an honest "we don't ship it") without ever reinstalling from the foreign repo.
  • Self-host swaps the catalog source, doesn't skip the audit. Detect self-host via the git remote (or a local skills/ holding these skills); then read the catalog from the local working tree and state the branch-vs-origin relationship, so a branch ahead of origin reads as "local is ahead," not as fleet-wide drift.

Definition of done

Each criterion is confirmed two ways: a file-check (the contract prose says it) and, where behavioral, a dual-executor probe (Opus in-session + gpt-5.5 via codex exec, per docs/patterns/probe-evals.md) that both executors pass against a fixture repo state.

  • Gap A — routing. SKILL.md and interview.md state a three-state routing keyed on prior-install evidence (not the block alone), and a probe presenting a "skills installed, no ## Agent skills block" repo state has both executors route to audit (not setup); audit-mode names a Missing map finding whose fix writes the block.
  • Gap B — .agents/ scan. Every "read what's installed" surface (audit-mode.md step 2, interview.md phase 1, SKILL.md phase 1) enumerates .agents/skills/ alongside .claude/skills/ at both project and global scope; a probe on a repo whose skill exists only under .agents/skills/ has both executors report that skill as installed (not "nothing installed").
  • Gap C — overlap taxonomy. audit-mode.md step 3's Overlap finding includes a cross-harness-duplication category that distinguishes a symlinked pair (note only) from divergent independent copies (flag + consolidate); a probe on a skill present in both .claude and .agents has both executors name the cross-harness category and correctly classify the symlinked case as benign.
  • Gap D — foreign source. audit-mode.md step 3 defines a Foreign-source finding keyed on skills-lock.json source ≠ asasher/asher-skills, advising per the pull-only invariant and never proposing a foreign reinstall; a probe on a repo with writing-great-skills from mattpocock/skills has both executors surface it as foreign-source and offer our equivalent-or-honest-no, not a reinstall from mattpocock.
  • Gap E — self-catalog. audit-mode.md step 1 handles self-host: catalog source is the local skills/ tree when the repo is asasher/asher-skills, with branch-vs-origin stated; a probe auditing this repo has both executors use local skills/ as the catalog and not report the local branch's own skills as "new in repo" drift.
  • No regression & no runtime. The greenfield case (no block, no installed skills) still routes to setup; no version stamp / hash / vNN marker is introduced (the no-stamp invariant holds); and the change adds no script — the skill stays prompt-driven. The setup skill's existing probes in evals/ still pass.

How it will be checked

Seam 1 — the contract prose. Grep/read each changed file for the required clause (routing predicate, harness-dir enumeration, each new finding). Cheap, exact, catches "did the words land."

Seam 2 — dual-executor probes. The load-bearing seam. New/updated scenarios in skills/setup-asher-skills/evals/probes.md + answer-key.md, one per behavioral gap (A–E), each driven through both an in-session Claude executor and gpt-5.5 via codex exec and graded against the answer key. This repo is a ready-made fixture: it exhibits A (no block, has skills), B/C (symlinked .claude→.agents), D (writing-great-skills foreign), and E (is the source). Left deliberately unchecked: no automated CI exists (§ environment) — every check is agent-driven on demand; that's the repo's accepted posture, not a gap this plan closes.

What's required

  • Edited contract files: SKILL.md, reference/audit-mode.md, reference/interview.md.
  • New/updated probe scenarios + answer key under skills/setup-asher-skills/evals/, one behavioral probe per gap A–E.
  • Evidence: the dual-executor transcripts and a pass/fail table mapping each probe to its answer-key criterion (per environment.md § Driving the app).
Risks & mitigations
  • Over-broad routing predicate — treating a stray unrelated skills/ dir as an asher-skills install and mis-routing to audit. Mitigation: the predicate requires evidence traceable to this repo (lock-file source, or the remote being asasher/asher-skills), not just any skills dir.
  • Symlink double-count — reading .claude and .agents naively reports every skill twice on this repo. Mitigation: dec-3 resolves symlinks before comparing; dec-4's benign-symlink classification is exercised by the gap-C probe.
  • Prose-only change that doesn't move behavior — reworded contract, unchanged model behavior. Mitigation: every behavioral criterion is gated on a dual-executor probe both models must pass, not on the prose alone.
  • Scope creep into setup's install path — gap D could tempt auto-removing foreign skills. Mitigation: dec-5 fixes it as surface-and-advise only; removal stays the user's decision.
Out of scope
  • Actually running a setup/audit against this repo to fix its real drift (missing block, 3-of-24 registered, broken backlog closure) — the issue defers that to groom; this plan hardens the skill, it doesn't remediate the repo.
  • Introducing any version stamp, hash, or drift-token mechanism — explicitly forbidden by audit-mode's no-stamp invariant.
  • Adding a runtime, server, or installer script — the skill stays prompt-driven.
  • Changing the closure rules, catalog, or the pull-only invariant itself (only adding an audit-side finding that enforces it).
