Ship a skill-authoring baseline pack from validated practice

A repo that authors prompt-driven skills currently receives software verification and evidence defaults, even though its real runtime is an executor harness and its honest proof is a graded probe transcript.

#35enhancementplan v1

The resolved install

A sixth domain overlays four validated skill-authoring baselines onto common, then fills only unresolved required steps with flagged software stand-ins.

Plan-or-skip

PLAN. The project threshold triggers because this crosses more than one subsystem/public interface: setup domain routing, the common policy contract, a new template pack, installed-copy reconciliation, and behavioral probes. The partial-pack fallback is also a new commitment whose semantics must be settled before editing.

Stories

  • As the owner of a repo that authors skills, I want setup to recognize that domain from its layout, so that I confirm an informed default rather than select a misleading software baseline.
  • As a skill author, I want verification framed as probes through real executor harnesses, so that the playbook tests the prompt-driven behavior I actually ship.
  • As a debugger, I want a skill bug reduced to a failing probe or confirmed transcript and its load-bearing prompt/reference line, so that the feedback loop targets the real decision error.
  • As a reviewer, I want transcripts and per-criterion verdicts, plus rendered proof for visual outputs, so that evidence is honest and gradable.
  • As backlog setup, I want a partial pack to resolve every required playbook without disguising software defaults as native skill-authoring practice.

The four native baselines

Pack fileDistilled generic baseline
verifying.mdPreserves the role header and the sections downstream reads: Checks, CI merge gate, Acceptance criteria. Checks become situated probes graded against an answer key written before any run; the same probes run through an in-session executor and an independent CLI executor; both must cite the deciding text, produce transcripts, and be graded per criterion. It says explicitly that executor probes replace “drive the app” for skill behavior, while scripts are driven directly and any real CI is still recorded.
environment.mdPreserves branching/deploy, local running, isolation, seed, auth, driving/evidence, presenting, staffing, and parallelism sections. There is no assumed app or stack: exercising a skill means driving a probe through an executor harness; files plus stdlib scripts make worktree isolation complete; seed state is the situated scenarios in each skill's evals/; executor roles and transcript capture replace app drivers. Presentation and staffing remain sibling-owned placeholders.
diagnosing-bugs.mdKeeps the feedback-loop → minimise → hypothesise → instrument → regression → cleanup phases, but leads with a failing situated probe or a confirmed misbehaving-run transcript. Reproduce model-dependent behavior across the two executor roles, then minimise the scenario to the prompt input and exact SKILL/reference line that causes the wrong decision; the resulting probe becomes the regression check. Script bugs still use a direct failing invocation.
evidence.mdPreserves the role header plus What to capture, Format/storage, and Presentation contracts. Default proof is the executor transcript and a pass/fail table mapping every probe to its prewritten answer-key criterion; behavioral reworks show the relevant before/after verdict shift. When a skill emits a visual artifact, add rendered screenshots or a short flow artifact. No repo hostnames, identities, model names, or local paths are baked in.

Key decisions

  • Register skill-authoring as a sixth work domain in setup and the common policy. A fresh repo with SKILL.md files under skills/<name>/ defaults to it; setup still confirms the inferred value, ambiguous evidence still defaults to software, and re-runs never silently change a recorded domain.
  • Ship exactly four native files and define one resolved scaffold set. environment, verifying, evidence, and diagnosing-bugs are IN because the issue names them and the repo has dogfooded source. implementing, refactoring, change-reviewer, and change-fixer are OUT as native files: their software baselines are substantially code-flavored, and the issue provides no validated skill-authoring source from which to distill replacements. The resolved set is common overlaid by the domain pack, then any absent member of the seven required step-playbook names is filled by its same-name software baseline. Setup step 1 inventory/reconciliation, step 2 scaffolding/completion, and run.md preflight all use that resolved set, so fresh installs and re-runs cannot make fallback playbooks invisible.
  • The common-overlay shadow rule is sufficient for environment.md. Setup already says the scaffold set is templates/common/ overlaid by templates/<domain>/ and “a pack file with the same name shadows the common file,” so the skill-authoring environment wins without a special case.
  • No fifth probe-evals.md template ships. Its reusable technique is distilled into verifying.md; adding an unreferenced project playbook would expand setup's contract beyond the issue's four named files.
  • Generalize roles, never this repo's bindings. Say “in-session executor,” “independent CLI executor,” “the staffing skill/playbook,” and “the skill's evals/”; omit tailnet URLs, account/repo names, named models, and machine-specific paths.
  • Append a pre-keyed issue-35 probe set to backlog's existing eval file. Three situated probes cover fresh skills-repo registration/install, same-name shadow plus partial fallback, and a mid-verify answer-key/dual-executor decision. Each run requires exact citations and ambiguity reporting; both executor roles must pass every probe.

Fresh install: before scaffolding anything, ask the user "what kind of work does this repo track?" with exactly five options: software / writing / research / ops / general. Default the answer from repo evidence and confirm rather than interrogate, in the same posture as step 3's binding questions: a code repo (package.json, src/, a test runner) defaults to software; a manuscript defaults to writing; ambiguous evidence defaults to software.

Fresh install: before scaffolding anything, ask the user "what kind of work does this repo track?" with exactly six options: software / skill-authoring / writing / research / ops / general. Default the answer from repo evidence and confirm rather than interrogate, in the same posture as step 3's binding questions: a skill-authoring repo (SKILL.md files under skills/<name>/) defaults to skill-authoring; a code repo (package.json, src/, a test runner) defaults to software; a manuscript defaults to writing; ambiguous evidence defaults to software.

Make the partial-pack rule coherent across setup and run

Current step-1 inventory sentence:

Compare the installed playbooks against the skill's scaffold set for that domain: templates/common/ overlaid by templates/<domain>/, with a same-name pack file shadowing the common file. Identify which scaffold-set playbooks lack a counterpart.

Planned definition used by step 1, step 2, and run:

Resolve the scaffold set in two passes: overlay templates/common/ with templates/<domain>/, with a same-name pack file shadowing common; then, for each required step playbook absent from that overlay, add the same-name templates/software/ baseline as a flagged stand-in. The required step names are implementing, verifying, evidence, diagnosing-bugs, refactoring, change-reviewer, and change-fixer.

Update step 1's comparison and completion criterion, step 2's “Scaffolding and reconciliation draw from…” / “Missing → write…” / completion lines, and reference/run.md's preflight completion criterion to say resolved scaffold set. For skill-authoring that is eleven counterparts: three unshadowed common files + four native pack files + four flagged software stand-ins.

How it will be checked

The behavioral seam is the dual-executor probe-eval discipline in docs/agents/probe-evals.md, not a conventional test suite: answer key first, the same situated probes through a Claude subagent and codex exec --sandbox read-only, citations required, transcripts graded into verdict tables. Static seams are directory listings, targeted greps, and source-versus-installed-copy comparison. There is no running app to drive and no screenshot requirement unless a changed skill emits a visual artifact.

What's required

  • The Claude-subagent and read-only-codex exec transcripts plus the P1–P3 verdict table against the prewritten answer key.
  • A static-check transcript for pack contents, forbidden-binding greps, six-option coherence, partial-fallback wording, and source/installed-copy equality.
  • Rendered evidence only if implementation changes or produces a visual artifact; the four markdown baselines themselves need no screenshot.
Risks
  • A partial pack silently drops required playbooks. Mitigation: setup resolves missing step files explicitly after overlay, flags each fallback, reports every gap, and P2 exercises the exact case.
  • A fallback is mistaken for validated skill-authoring practice. Mitigation: only the four proven files are native; every software remainder carries the code-flavored stand-in header and appears in the setup report.
  • The environment shadow is missed because environment was classified common. Mitigation: the existing same-name shadow rule is called out in setup and tested directly in P2/ac-12.
  • The baseline leaks this repo's machine, roster, or identity. Mitigation: generic-role wording plus the grep-checkable ac-7 exclusion list.
  • An answer key is reverse-engineered from executor output. Mitigation: probes and key land together before runs; transcripts record the run order.
Out of scope
  • A writing/docs pack — deferred until a real project validates its practice.
  • Native skill-authoring versions of implementing.md, refactoring.md, change-reviewer.md, or change-fixer.md; they remain explicit stand-ins until validated source exists.
  • A fifth probe-evals.md project playbook; the reusable discipline is inlined into the verifying baseline.
  • Changing backlog's issue routing, setup's recorded-domain re-run rule, or any domain other than registering the sixth option and updating fallback accounting.
  • A conventional automated test suite or live setup run against another repo; the situated probes are the project-recorded verification seam.
