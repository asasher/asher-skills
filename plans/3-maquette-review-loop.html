Retrofit maquette to consume the review-loop primitive

Maquette's human sign-off gates reinvent an approve-in-chat loop in prose; they should defer to the shared review-loop skill so the same annotate→verdict→approve surface serves a maquette as it serves a plan.

issue #3enhancementplan v1

What this actually is

Maquette carries no server and no scripts — its "presentation loop" is informal gate prose ("write BRIEF.md, read it back to the user, the intake ends when the user approves"). That prose reinvents an annotate→revise→approve loop in words. This change deletes that reinvented prose and points the two human sign-off gates at the review-loop skill by plain name. Nothing is copied out of review-loop; maquette imports none of its files.

Implementation decisions

  • The reviewable artifact is a rendered HTML view of the deliverable, not the markdown. BRIEF.md / JOURNEYS.md stay the canonical repo deliverables (HANDOFF and the pipeline reference them by name). At each gate maquette renders the current deliverable to a self-contained HTML page whose sections/journeys/screens each carry a stable id, and serves that. Forced by review-loop: it binds approval to the served HTML's content hash and anchors annotations to stable ids — markdown has neither.
  • Point at review-loop by plain name, never by copying its scripts. Maquette imports no review-loop files; it invokes the skill's serve/await per review-loop's own reference/scripts.md. This is the decomposition contract — review-loop is a root primitive, consumers reference it by name.
  • The three verdicts map onto maquette's gate semantics. approve → proceed to the next phase; approve_with_nits → apply notes and proceed without a re-review round; request_changes → revise the deliverable, write a ledger disposition per annotation, re-serve, re-await. This is maquette's old "corrections loop back into questions," now mechanized.
  • Gated scope = the two existing human sign-off gates only. BRIEF (phase 1) and JOURNEYS (phase 3). The dead-click sweep (phase 6) stays an agent-run internal gate, not a human review — untouched. HANDOFF (phase 7) has no human gate today; left as-is (a future opt-in review at most).
  • Surface config is inherited, not defined. Maquette relies on the repo's docs/agents/environment.md § Presenting (the tailnet surface) exactly as review-loop's dependency surface specifies; absent a surface config, review-loop degrades to local-only. Maquette adds no surface knowledge of its own.

Definition of done — acceptance criteria

  • SKILL.md phase-1 and phase-3 gates name the review-loop skill as the approve surface (serve→await→branch on verdict); neither gate still instructs "read it back in chat and wait for the user to say approve" as the mechanism.
  • intake.md Exit gate routes brief sign-off through review-loop and expresses the corrections loop as request_changes → revise + ledger → re-serve, preserving "no work on an unapproved brief."
  • Exactly one maquette reference documents the composition (render → HTML with stable ids → review-loop serve --kind maquette → await exit-code branch → ledger on revision) and states maquette carries no review server of its own and imports no review-loop files.
  • grep -rn "review-loop" skills/maquette shows matches only as a by-name sibling pointer; no copied review-server.py, review-await.py, or chrome.* exists under skills/maquette/.
  • No reinvented-loop prose remains: maquette no longer describes its own annotate/verdict/approve mechanics independent of review-loop.
  • End-to-end dry-run: a sample BRIEF rendered to HTML and served via skills/review-loop/scripts/review-server.py --kind maquette --issue <slug> yields a review URL on the tailnet surface + a hub entry, and review-await.py gates on the verdict. Headless-Chrome screenshotting is SIGKILLed here, so visual verification is substituted by a curl/DOM transcript (chrome injected, stable ids present, /version hash endpoint live) — disclosed as an environment gap, matching #2's evidence handling.
  • maquette's evals assert the gate-through-review-loop behavior — at least one probe expects the model to reach for the review-loop skill by name at a sign-off gate — and the build-eval hard gate treats gate approval as a review-loop approve verdict.
  • No regression: intake grilling, research, data design, build, demo hardening (dead-click sweep intact), and handoff are unchanged except for the gate mechanism; BRIEF.md and JOURNEYS.md remain the named canonical deliverables.

Risks
  • The markdown→HTML render step is new surface. Keep it a documented convention (a stable id per section), not a script, so maquette does not grow its own machinery. If it later needs code, that is a follow-up issue.
  • Two forms of the deliverable (canonical .md + rendered .html for the gate) risk drift. Mitigate by rendering at gate time from the current .md and treating the .html as ephemeral — not a committed deliverable.
Out of scope
  • Building or copying any server into maquette.
  • Changing review-loop itself — #2 shipped it; this issue only consumes it.
  • Gating HANDOFF.md or the dead-click sweep through review-loop.
  • Running the full build-eval tier.
