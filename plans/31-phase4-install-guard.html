setup-asher-skills Phase 4 — guard the self-host write path & verify each install landed

Running setup-asher-skills against its own repo shredded the catalog, and a silently-failed install shipped a broken skill closure — both because Phase 4 trusts npx skills add too much.

#31enhancementplan v1contract-only

Problem

The real dogfood (2026-07-10) proved the routing/audit logic sound but exposed the delegated install layer (Phase 4, driving skills CLI v1.5.15) as unsafe in two ways:

  • HIGH-1 — self-host clobber. When the target repo is asasher/asher-skills, npx skills add treated the repo's own skills/<name>/ source dirs as install targets, moved them into .agents/skills/ and replaced each with a symlink — corrupting the source of truth, silently.
  • HIGH-2 — silent miss. npx skills add --skill plan prints ✖ No matching skills found and exits 0, installing nothing; a real run then ships backlog with a broken closure. -y mode reported "Installed 3 skills" for 4 requested, with no error.
  • LOW — hash gap. A self-host fallback entry can't recompute skills-lock.json computedHash (not a plain sha256), so a later reconcile misreads it as drift.

The fix is prompt/contract hardening only — edits to setup-asher-skills's reference and eval files. The skill ships no runtime; it just needs to stop trusting the CLI blindly.

The shape — Phase 4, per skill in the closure

Two new gates wrap every install: a self-host guard before, and landing verification after — with a single self-host fallback serving both the guarded case and a verified miss.

Stories

  • As the maintainer of asher-skills, I want setup run against my own repo to leave skills/ untouched, so that self-configuring the catalog never corrupts the catalog.
  • As a user setting up a normal project, I want a skill that npx skills add failed to install to be caught and installed another way, so that I never ship a composer with a silently-broken sibling closure.
  • As a user re-auditing later, I want a self-host fallback entry to read as expected rather than drifted, so that reconcile doesn't flag noise I can't resolve.
  • As the next agent editing setup, I want the "repo is the source" test to live in one place, so that the READ and WRITE paths can't drift apart again.

Key decisions

  • Reuse, don't duplicate, the self-host detection. Lift audit-mode.md step 1's "repo is the source" test (git remote asasher/asher-skills, or a local skills/ holding these skills) to a single named notion referenced by both the audit READ path and Phase 4's WRITE path. Two divergent copies are exactly how the READ path ended up guarded while the WRITE path stayed exposed.
  • Guard by skill-source, not just repo identity. The write-guard skips npx skills add for any skill whose install source would be the repo's own skills/<name>/. Per-skill and source-based, so it holds even if only some closure members are repo-owned — the source dir is never an install target.
  • Verify landing on the filesystem, never the exit code. After each install, confirm <name> is actually present via the install dir and/or its skills-lock.json entry. The contract states explicitly that npx skills add exits 0 on a no-match and under-reports counts under -y, so the exit status is not evidence.
  • One fallback mechanism — self-host placement — for both cases. When the tool won't (miss) or shouldn't (self-host repo) install, place the skill's files directly: from the repo's own skills/<name>/ in the self-host case, else fetched from the asasher/asher-skills endpoint. Never break the pull-only-from-this-repo invariant.
  • Accept the computedHash gap explicitly. A hand-placed entry can't reproduce skills-lock.json's computedHash. Rather than fabricate a wrong hash that reads as drift, mark the entry's origin as fallback and have audit-mode treat a fallback-origin entry as expected, not drift. Degrade gracefully; note the limitation to the user.
  • Contract-only, no runtime. All changes are to setup-asher-skills's reference/interview.md, reference/audit-mode.md (and a light SKILL.md mention), plus two new probes in evals/. No scripts — the skill drives npx skills, it does not wrap it in code.

Definition of done

static = confirmed by reading the changed contract text · behavioral = confirmed by a dual-executor probe (Opus in-session + gpt-5.5 via codex exec), passing only if both executors pass.

  • static interview.md Phase 4 states a self-host precondition that gates the npx skills add write path, and names the same "repo is the source" detection as audit-mode.md step 1 (git remote asasher/asher-skills, or a local skills/ holds these skills).
  • static The contract states that in the self-host case a skill whose source is the repo's own skills/<name>/ is never passed to npx skills add — the source dir is never an install target — and that the map, pointer, and playbooks are still written.
  • static Phase 4 requires per-skill landing verification after each install by checking the install dir and/or the skills-lock.json entry for <name>, and states explicitly that npx skills add can exit 0 having installed nothing and can under-report counts under -y, so the exit code is not trusted.
  • static On a verified miss, Phase 4 specifies the self-host fallback — place the skill's files directly (from the repo's own skills/<name>/ when self-host, else fetched from the asasher/asher-skills endpoint) — rather than trusting the failed install, without violating pull-only-from-this-repo.
  • static The computedHash limitation is documented: a self-host fallback entry cannot recompute skills-lock.json computedHash by hand; the contract says to mark the fallback origin (not fabricate a hash) and audit-mode treats a fallback-origin entry as expected, not drift.
  • static The self-host detection exists as one shared, named notion referenced by both audit-mode.md (READ) and interview.md Phase 4 (WRITE) — not two independently-worded copies.
  • behavioral A new probe (self-host write) is in evals/probes.md with an answer-key.md entry: an executor invoked with setup on the asasher/asher-skills repo itself must guard the write path — emit no npx skills add against the repo's own skills/ — while still writing the map/pointer, citing the file and sentence. Both executors pass.
  • behavioral A new probe (silent miss) is in evals/probes.md with an answer-key.md entry: given npx skills add --skill plan exiting 0 with nothing landed, the executor must detect the miss via filesystem landing verification (not the exit code) and fall back to self-host placement, citing the file and sentence. Both executors pass.

How it will be checked

Two seams, the fewest that prove the criteria:

  • Static — the contract text. AC-1…6 are read directly against the four changed files in skills/setup-asher-skills/. The skill is its prose, so the text is the artifact under test.
  • Behavioral — the probe harness. AC-7…8 run the two new probes through the existing dual-executor design (docs/patterns/probe-evals.md, environment.md § Model staffing checker role): an Opus subagent in-session and gpt-5.5 via codex exec -s read-only, each pointed at skills/setup-asher-skills/, graded against answer-key.md. A probe passes only if both executors pass.

Deliberately unchecked: the actual skills CLI runtime behavior (v1.5.15 bugs) — out of our control, hardened around, not fixed here; and the exact skills-lock.json schema — verification is described by behavior (an entry for <name> exists) so it survives minor CLI-version drift.

What's required on completion

  • A prose read of the four changed files confirming AC-1…6, and the two probe transcripts (both executors, each probe) with a pass/fail table mapping each behavioral criterion to its answer-key entry.
Risks & mitigations
  • The self-host repo's catalog skills stay only in skills/, not a harness-discoverable dir. That is the intended state — the guard's job is to not move source. Wiring individual catalog skills into .claude/skills/ for the repo's own harness use is a deliberate, separate choice, called out as out of scope. Mitigation: the guard still writes the ## Agent skills map, so the map is correct even when discovery is handled elsewhere.
  • skills-lock.json schema is the CLI's, not ours. Mitigation (dec-3): verification keys on "an entry for <name> is present" behaviorally, not an exact field layout.
  • Over-triggering the guard could skip a legitimate install in a repo that merely has a skills/ dir. Mitigation (dec-2): the guard is per-skill and source-based — it only skips a skill whose install source is the repo's own skills/<name>/, not any repo with a skills/ folder.
Out of scope
  • Filing the upstream skills CLI bug (the plan-token no-match + exit-0, and the -y under-count). A follow-up action the issue notes, not a contract change here.
  • Fixing the skills CLI itself. We harden setup around it.
  • Wiring the self-host repo's catalog skills into .claude/skills/ for its own harness use — a deliberate maintainer choice, not setup's job.
  • Any change to routing or the three-surface audit (#7, #27) — those are sound; this is only the delegated install layer.
