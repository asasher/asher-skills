Add a draft work-type for judgment-terminal work

Backlog's three work-types all assume a testable spec, so work whose correctness is taste or fit — a memo, copy, a research synthesis, code docs — has no honest home and gets forced through a mechanical verify gate that cannot judge it.

#33enhancementplan v1

The draft branch

A fourth work-type that routes to produce → review-loop → verdict-as-done: the human review verdict is the definition of done, and the artifact is kept.

Two comparisons that fix the boundary

draft is enhancement-shaped but judged differently, and it is the opposite of prototype on artifact fate.

Stories

  • As Asher grooming the backlog, I want to label a "write the launch memo" issue draft, so that it isn't forced into enhancement and judged by a mechanical verify it can never pass.
  • As an issue thread working a draft, I want the route to send me to produce → review-loop → verdict, so that I stop at the human verdict instead of inventing testable acceptance criteria for a taste call.
  • As an issue thread, I want draft to keep the artifact (commit + merge), so that the memo/doc I produced ships — unlike prototype, which would delete it.
  • As a reader of the policy, I want the draft role to name code-docs as a fit, so that documentation work routes here too.
  • As a maintainer verifying the change, I want a dual-executor probe, so that two differently-modeled agents demonstrably route a judgment-terminal task to draft and name the verdict as done.

Key decisions

  • Add draft as a fourth work-type role, not a readiness or a flavor of enhancement. Work-type decides how the agent works an issue; judgment-terminal work is a distinct how (verdict-terminal, no mechanical verify), so it earns its own row in backlog-policy.md § Label roles → Work-type, next to bug/enhancement/refactor.
  • The role text pins the verdict-as-done and the keep-the-artifact rule. Definition: "produce a novel artifact whose correctness is taste/fit; done is the human review verdict at the review gate, with no mechanical verify pass/fail; the artifact is kept (committed and merged) — unlike prototype, which deletes it." The prototype contrast lives in the role text itself so the two never blur.
  • Route draft inline in issue-loop.md step 3 — no new heavy branch reference. The branch is thin (produce via staffing → present via review-loop by name → keep), so it is a bullet alongside bug/enhancement/refactor rather than a new reference/draft.md. It composes review-loop and staffing by name, importing neither.
  • Verify degenerates, stated in issue-loop.md step 4. For a draft, step 4 is skipped or collapses to "the review gate returned approve" — there is no acceptance-criteria run because there is no testable spec. Step 6 adversarial review likewise degenerates for a pure-prose artifact (the human verdict was the real review); code-docs may keep a light correctness pass.
  • Edit the template and the installed copy, in lock-step. Both skills/backlog/templates/backlog-policy.md (source template) and docs/agents/backlog-policy.md (installed), and both skills/backlog/reference/issue-loop.md (source) and .claude/skills/backlog/reference/issue-loop.md (installed), so a future backlog setup re-install carries draft and the checked-in policy already has it.
  • Grooming gets a one-line recognizer. A note under the work-type roles / readiness guidance on how to spot a draft ("the deliverable is an artifact judged by taste/fit — a memo, copy, a synthesis, code docs — with no testable spec"), so groom assigns it instead of defaulting to enhancement.
  • The dual-executor probe lives in a new skills/backlog/evals/probes.md (the backlog skill has no evals/ yet), following docs/patterns/probe-evals.md: a judgment-terminal scenario + answer key, run once through an in-session Claude executor and once through gpt-5.5 via codex exec.

Definition of done

  • backlog-policy.md § Label roles → Work-type lists a draft role as a fourth bullet beside bug/enhancement/refactor, present in both the installed docs/agents/backlog-policy.md and the source skills/backlog/templates/backlog-policy.md.
  • The draft role text states its definition of done is the human review verdict at the review gate, and states explicitly that there is no mechanical verify pass/fail.
  • The draft role text states the artifact is kept (committed and merged) and names the contrast with prototype (throwaway — keep the answer, delete the artifact).
  • The draft role text names code-docs as an in-scope use.
  • issue-loop.md step 3 lists a draft route that (a) produces the artifact via the staffing skill by name, (b) presents it via the review-loop skill by name, and (c) treats the verdict as done and keeps the artifact — present in both the installed and source issue-loop.md.
  • issue-loop.md step 4 (verify) states that for a draft the step is skipped or degenerates to "the review gate passed" — no mechanical acceptance-criteria run.
  • Given a judgment-terminal scenario, an in-session Claude executor routing per the edited policy classifies the work-type as draft, names the human review verdict as its definition of done (not a mechanical verify pass), and states the artifact is kept (not deleted).
  • The same scenario run through gpt-5.5 via codex exec (the second, independent executor) reaches the same three conclusions — draft, verdict-as-done, artifact-kept — matching the probe's answer key.
  • The probe scenario and its answer key are committed under skills/backlog/evals/probes.md, and neither executor conflates draft with prototype (i.e. neither proposes deleting the artifact).

How it will be checked

Doc-shape criteria (AC-1..6) are confirmed by reading the four edited files. The routing criteria (AC-7..9) are confirmed at the skill's behavioral seam: the dual-executor probe from docs/patterns/probe-evals.md — one in-session Claude run (Agent tool) and one codex exec -s read-only run, each fed the judgment-terminal scenario and graded against the answer key. No mechanical CI exists in this repo; every check is agent-driven on demand. Deliberately left unchecked: the aesthetic quality of any real draft artifact (that is exactly the human-verdict surface this work-type introduces, not something to automate).

What's required

  • The two probe transcripts (Claude in-session + codex exec) plus a per-criterion pass/fail verdict table mapping AC-7..9 to the answer key.
  • A screenshot is not required — the artifacts here are markdown edits, not a rendered surface; the probe transcripts are the evidence.
Risks
  • Executors conflate draft with prototype (both "produce an artifact, review it"). Mitigation: the role text and the tables lead with the artifact-fate contrast (kept vs deleted), and AC-9 fails the probe if an executor proposes deleting the artifact.
  • Degenerate steps read as "skip everything," letting a draft merge unreviewed. Mitigation: step 4/6 degenerate because the review-loop human verdict already ran in step 3 — the gate moves, it does not vanish. The wording ties "verify passed" to "the verdict was approve."
  • Template/installed drift — editing one copy but not the other. Mitigation: AC-1 and AC-5 require both copies; dec-5 makes lock-step explicit.
  • Grooming keeps defaulting to enhancement and draft is never assigned. Mitigation: the dec-6 recognizer line; the probe exercises the classification step, not just the route.
Out of scope
  • A new reference/draft.md branch file — the route is thin enough to inline (dec-3). Revisit only if the branch grows its own procedure.
  • Reworking verify.md / adversarial-review.md internals — draft is handled by the routing note in issue-loop.md, not by rewriting those references.
  • Retro-adding evals/ to other skills — this plan adds only the backlog probe needed to verify draft.
  • Building the review surface or changing review-loop / staffing — draft composes them by name, unchanged.
  • Migrating existing backlog issues to draft — grooming does that going forward, per-issue.
