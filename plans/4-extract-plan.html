Extract plan into a standalone thin-composer skill

The planning step — decide whether a plan is warranted, settle blocking design questions, write a reviewable plan artifact, and hold it at an approval gate — is buried inside backlog and coded for a dev branch. Any effort worth planning (a research sprint, an ops change, a product bet — not only code) wants it, so it should be an extracted thin composer: it produces the plan artifact and borrows sign-off from review-loop and role selection from staffing, both by name.

issue #4enhancementplan v1

Approach

Unlike review-loop (which owns machinery — a server, scripts, a hub), plan owns no runtime of its own. It is a thin composer: it authors an HTML plan and then delegates — presentation and the annotate→revise→approve gate to review-loop by name, and "who should author/build this" to staffing by name. The extraction lifts backlog's reference/plan.md gates, templates/planning.md format, and plan-skeleton.html into a standalone skill shaped like the shipped staffing/review-loop, genericizes the dev-isms, and strips the dev-only tail (the "commit the plan to the branch / implementing agents read it from disk / posterity tracker-write" steps belong to backlog, not a generic plan skill). It is deliberately non-breaking: backlog keeps its own bundled plan step unchanged; rewiring it to consume this skill by name is a separate issue.

The gates it keeps — and the tail it drops

The generic composer runs four gates and stops at Approved. backlog's dev-only 5th gate (commit + hand to implementation) is not the plan skill's job.

File layout

New skill under skills/plan/, mirroring staffing/review-loop's shape. Nothing is deleted from backlog. Note the absence of a scripts/ dir — the composer ships no runtime.

User stories

  • As anyone planning an effort (a research sprint, an ops migration, a product change — not only code), I want to invoke plan by name and get a reviewable plan artifact with explicit acceptance criteria, so that I think before I build without adopting a dev workflow.
  • As a skill author (backlog, goodwork, a research skill), I want to compose plan by name to produce and get sign-off on a plan, so that I don't fork the planning gates into my skill.
  • As an agent, I want the composer to present the plan through review-loop and block on its verdict, so that I proceed only on an explicit human approve and revise on request-changes — without the composer re-implementing a review server.
  • As an agent, I want "who should author or build this" answered by staffing by name, so that role selection stays consistent with the machine's roster instead of being hardcoded in the plan skill.
  • As a user with a tiny, low-risk change, I want the composer to skip planning per a stated threshold, so that planning is applied where it pays and not as ceremony.
  • As a user with no project playbook and no review surface, I want the skill to still run on bundled defaults and fall back to opening the rendered plan locally, so that it is usable outside this repo.
  • As a Codex user, I want the skill to present via agents/openai.yaml, so that it works when installed into Codex.

Implementation decisions

  • Standalone skill at skills/plan/, shaped like staffing/review-loop: SKILL.md + README + reference/ + templates/ + agents/openai.yaml + evals/. No scripts/ — the composer ships no runtime.
  • Thin composer, delegation by name. The approval gate says "present via review-loop"; role questions say "resolve via staffing." No review-server.py/review-await.py is copied in, and no path-import of another skill's files — composition is by plain name only, per AGENTS.md § the composition model.
  • Contract split into two references. reference/plan-contract.md carries the four gates, the plan-or-skip threshold, and the two compositions (review-loop for gate 4, staffing for authorship). reference/authoring.md carries what a plan covers and the house format (visual-first, stable ids, ac-N criteria, self-contained HTML) — the domain-neutral rewrite of backlog's planning.md § Format/What-a-plan-covers.
  • Strip the dev-only tail. The bundled contract's terminal gate is Approved (the approval record). "Commit the plan to the work branch," "implementing agents read it from disk," and the posterity tracker-write are removed and explicitly labelled as the caller's post-approval concern — what backlog wraps around the composer.
  • Domain-neutral by default. The bundled contract and skeleton read for any effort. Acceptance criteria are "checkable pass/fail" without assuming a running app; test-seams / verify / evidence / repo-path framing is not in the bundled contract — it belongs to the project playbook.
  • Two-layer, standalone-usable (like staffing). Bundled references carry the full default contract (threshold, format, gates) so the skill works with no project playbook. The playbook (docs/agents/planning.md) is delta-only: it overrides the threshold, the plan-file location convention, and the approval authority. When no playbook and no review surface exist, the contract states the local fallback (open the rendered plan on the machine, take the verdict in conversation).
  • Authoritative plan skeleton. templates/plan-skeleton.html is the plan's own document shape (plan owns it). It carries stable section ids and <li id="ac-N" data-criterion> criteria, inline styles + inline SVG, no external fetches; the backlog-specific comment header ("backlog plans", templates/planning.md) is degeneralized.
  • Gate 2 is means-neutral. Blocking design questions are settled before writing "by whatever the caller has — a prototype, a spike, research," and the answer is folded into the plan. prototype is not a declared dependency of this skill (only review-loop and staffing are).
  • Three-part dependency surface declared in SKILL.md: bundled references (its own gates + format + skeleton) · project playbook (docs/agents/planning.md, delta-only) · sibling skills (review-loop = the sign-off gate; staffing = who authors/builds).
  • Non-breaking. backlog's reference/plan.md, templates/planning.md, and plan-skeleton.html are untouched; it still plans as today. Rewiring backlog to consume this skill by name (and deleting its copy) is a separate issue.
  • Staffing of this build: plan authorship (this artifact) is orchestrator work. Build-out delegates — the taste surface (SKILL/README/reference copy, skeleton degeneralization) to a Claude builder (opus-4.8/fable-5); any mechanical lift/format work to the backend builder (gpt-5.5 via codex exec). Probe checking runs on both a Claude and a gpt-5.5 executor.

Test seams

  • File/content checks (primary seam). ac-1 through ac-8 and ac-10 are grep/parse assertions over the shipped tree: layout, the three-part dependency declaration, the composition-by-name boundary (no copied scripts), the stripped tail, domain-neutral wording, the self-contained skeleton, valid openai.yaml, and backlog left untouched. A skill that ships no runtime is verified structurally, not by driving a process.
  • Probe eval (routing seam). The skill's real value — does an agent invoke it, honor the threshold, compose review-loop and staffing by name, and stop at approval — is judged by dual-executor dry-runs per docs/patterns/probe-evals.md (ac-9).
  • Deliberately untested: the review runtime (serve, annotate, verdict-coded await, hash-binding) — that is review-loop's surface, exercised by issue #2, and this skill only names it. Re-testing it here would test the wrong skill. Likewise the taste of the generated plan copy is covered by the human's own review pass, not automation.

Risks
  • Transitional duplication: backlog's plan step and the new plan skill coexist until the rewire issue lands. Mitigation: the deferral is noted; the duplication is intentional and tracked, not drift.
  • Over-thinning: stripping too much dev context could leave the generic contract unable to reconstitute a rigorous dev plan. Mitigation: the dev specifics live in the project playbook (docs/agents/planning.md) that the contract composes over, so a dev caller still gets the full rigor; the bundled default is the floor, not the ceiling.
  • Composition-by-name is a convention, not an enforced import — a careless caller could ignore review-loop/staffing. Mitigation: the probe eval (ac-9) is exactly the check that agents route to the siblings rather than reinventing them.
Out of scope
  • Rewiring backlog to consume plan by name and deleting backlog's bundled plan step — a separate rewire issue.
  • Extracting prototype as a sibling skill; gate 2 stays means-neutral and depends on no prototype skill here.
  • A setup command that installs the delta-only planning playbook into a fresh repo — for this repo the playbook already exists as docs/agents/planning.md.
  • Any change to review-loop or staffing themselves, or to the tailnet surface config in docs/agents/environment.md.
