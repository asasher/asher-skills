to-spec — synthesize a conversation into a spec

A long design conversation with the agent ends with a decision but no durable artifact; the next step (splitting it into pickup-able work) has nothing to read. to-spec captures the decided direction as a repo spec — pure synthesis, no re-interview.

issue #8enhancementplan v1

Where it sits

Root of the spec→tickets pair: to-spec turns the conversation into one high-level direction doc that to-tickets (#9) later splits. No interview — it captures what was already decided.

Dev vs non-dev — one template, sections gated

The skill classifies the work, then keeps or drops the dev-only sections. Our skills aren't all dev, so the template must serve non-dev direction docs too.

Spec template — adapted from Matt's, gated for non-dev

Adapted from Matt Pocock's to-spec and shipped as our own (invariant: we never install external skills). Core sections are always present; dev-only sections are skipped when they don't apply.

User stories

  • As an agent finishing a long design conversation, I want to synthesize what was decided into a spec doc, so that the direction is captured durably without re-interviewing the user.
  • As the person driving the session, I want the spec to record only what we actually decided, with gaps flagged, so that I'm not asked questions I already answered and open points are visible.
  • As to-tickets (#9), I want a predictable spec doc at a known path, so that I can split it into pickup-able tickets.
  • As someone writing a non-dev spec (a process, a piece of content, a decision), I want the dev-only sections skipped, so that the template fits my work.
  • As the person driving the session, but AFK, I want the spec presented for sign-off through the review surface, so that I can approve it from my phone; when I'm present, inline approval is enough.
  • As a maintainer, I want the spec to carry no file paths or code snippets, so that it doesn't rot as the codebase moves.
  • As a Codex user, I want the skill to present correctly when installed into Codex, so that it works across harnesses.

Implementation decisions

  • New standalone skill skills/to-spec/, laid out to match the shipped skills/staffing/ and skills/review-loop/ primitives: SKILL.md (command surface), README.md, agents/openai.yaml, reference/ (the synthesis method + template-section guide), templates/ (the fillable spec scaffold), evals/probes.md. Self-contained at the file level; composes siblings by name only.
  • Pure synthesis, no interview. SKILL.md and the synthesis reference mine the current conversation + codebase understanding and capture what's already decided; they forbid re-interviewing. Unresolved points are flagged in the spec's Notes, not turned into questions to the user.
  • Artifact = a repo doc at docs/specs/<name>.md (markdown), described as the high-level direction doc that to-tickets consumes. The skill scaffolds it from templates/spec.md.
  • Generic vocabulary. "spec" and "ticket" throughout — never GitHub-specific "issue." The pair is spec→tickets.
  • Dev/non-dev gating. The synthesis method classifies the work; dev specs include Testing decisions + Test seams and run the "sketch seams, prefer the highest existing seam" step; non-dev specs skip both. The template marks those sections optional.
  • No file paths / code snippets in the spec (they go stale); the single prototype-validated-snippet exception is preserved, mirroring the plan/PRD rule.
  • Optional review-loop for sign-off. Declared as an optional sibling: when the user is AFK, to-spec renders the markdown spec to a self-contained review HTML and presents it through review-loop (composed by name, per the presentation surface); when present, approval is inline. Skipping review-loop still yields a valid committed spec — no hard dependency.
  • Dependency surface declared in SKILL.md with all three categories: bundled references (synthesis method + spec template), project playbooks (the repo's spec-location / vocabulary convention — defaults to docs/specs/; presentation-surface config when review-loop is used), sibling skills (optional review-loop only).
  • Codex manifest. agents/openai.yaml per docs/patterns/codex-compat.md, allow_implicit_invocation: false (a synthesis operator skill that writes repo docs, not implicitly fired).
  • Pre-deployment probe eval evals/probes.md: situated dry-run probes with a pre-written answer key, dual-executor (Opus in-session + codex exec), per docs/patterns/probe-evals.md, covering the criteria below.

Definition of done — acceptance criteria

Each is checkable pass/fail: probe dry-runs against executor models (Opus in-session + codex exec) with correct-action-AND-citation grading, plus file/grep checks for structural ones.

  • Skill shape. skills/to-spec/SKILL.md frontmatter identifies a user-invocable synthesis skill (name: to-spec, user-invocable: true, a description that reads as pure conversation→spec synthesis); the directory mirrors the shipped primitives (SKILL.md, README.md, agents/openai.yaml, reference/, templates/, evals/); and no file in the skill imports or reads another skill's files (grep finds no cross-skill paths).
  • Dependency surface. SKILL.md declares all three pointer kinds — bundled references, project playbooks, sibling skills — and states sibling skills as optional review-loop (for AFK sign-off; else inline), with no other sibling dependency.
  • No interview. A probe in which the user left a point undecided resolves to: synthesize from what's decided and flag the gap in the spec's Notes — do not re-interview. SKILL.md / synthesis reference forbid re-interviewing, citably.
  • Artifact. The output is a spec written as a repo doc at docs/specs/<name>.md, described as the direction doc to-tickets consumes — not a ticket, not a GitHub issue.
  • Generic vocabulary. The skill and template use "spec" and "ticket," never GitHub-specific "issue" as the unit of work; a grep over the shipped skill confirms it.
  • Adapted template. The bundled spec template carries the section set — Problem, Solution, User stories, Implementation decisions, Testing decisions, Out of scope, Notes — with the dev-only sections (Testing decisions, Test seams) explicitly marked optional / skip-when-N/A, so it's usable for non-dev work.
  • Dev-only seams step. A probe on a non-dev spec confirms the "sketch test seams / prefer the highest existing seam" step and the dev-only sections are skipped; a probe on a dev spec confirms they're applied.
  • No stale content. The synthesis method states the spec carries no file paths or code snippets, with the single prototype-validated-snippet exception preserved; a probe asking whether to inline a file path resolves to "no, describe it in prose."
  • Optional sign-off. A probe with the user AFK routes sign-off through review-loop; a probe with the user present approves inline. The skill does not hard-depend on review-loop — skipping it still produces a valid committed spec.
  • Codex manifest. skills/to-spec/agents/openai.yaml parses and is well-formed per docs/patterns/codex-compat.md, with allow_implicit_invocation: false.
  • Probe eval ships. skills/to-spec/evals/probes.md exists with a pre-written answer key covering ac-1..ac-10, dual-executor, per docs/patterns/probe-evals.md.

Test seams

This ships a skill (a prompt-driven procedure), so the seam is the probe eval, not a code test — the highest existing seam for skills in this repo (per environment.md § Driving the app and docs/patterns/probe-evals.md). Each acceptance criterion is exercised by at least one situated dry-run probe against both executors, graded correct-action-AND-citation, with structural criteria (ac-1, ac-5, ac-10) additionally confirmed by file/grep/YAML-parse check. No new script or runtime is introduced, so there is nothing else to seam.

Evidence required

  • The probe-eval transcript from both executors (Opus in-session + codex exec) with a pass/fail verdict table mapping each probe → its criterion.
  • A grep/YAML-parse check log for the structural criteria (cross-skill imports, "issue" vocabulary, openai.yaml well-formedness).
  • A sample spec produced from a real conversation (this decomposition is the canonical input) to show the template renders end-to-end for a dev case, and one non-dev case to show the gating.
Risks
  • review-loop serves HTML with stable ids, but the spec is markdown — the AFK path needs a render step (dec-7). Kept optional so the core path (inline approval) never blocks on it.
  • "Adaptable to non-dev" can drift into vagueness. Mitigation: the template ships concrete non-dev guidance and a probe (ac-7) grades the gating both ways.
  • Overlap with the existing plan playbook (both synthesize direction). Distinction held explicitly: a spec is a coarser, pre-ticket direction doc; a plan is per-issue and gated for approval before implement.
Out of scope
  • to-tickets (#9) — the consumer that splits a spec into tickets. Separate issue; this plan only fixes the spec's shape and path as its contract.
  • Any interview / elicitation flow — explicitly excluded; to-spec is pure synthesis.
  • Installing or vendoring Matt Pocock's skill — we adapt and ship our own; no external skill is installed.
  • Changes to the backlog loop, staffing, or review-loop skills themselves.
