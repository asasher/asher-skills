Hold the review-await watch on a cheap dedicated watcher subagent

When a thread pauses at a human-approval gate, something must hold the watch on the review. Today the capable orchestrator holds it inline — so it either rightly abandons the wait to save tokens, or a single review-await.py --timeout / Bash-tool ceiling drops a long AFK review. The verdict then lands server-side but never reaches the agent, and we fall back to approving in chat or hand-polling events.jsonl.

issue #16enhancementabsorbs #23plan v1

Approach

Move the watch off the orchestrator and onto a dedicated watcher subagent whose only job is to hold it — so it never "rightly abandons" the wait (holding it is its whole task) and it costs almost nothing to park. Its model is a staffing decision, not a hardcoded "sonnet": review-loop composes the staffing skill by name to route a watcher task to the cheapest reachable floor-class model — which differs by harness. The watcher loops-until-verdict, re-arming review-await.py after each timeout, so no single timeout ceiling can drop a long wait. On a real verdict the subagent completes, and its completion carries the verdict back to the parent, which resumes the gate — without manually polling events.jsonl (the fix that absorbs #23). The change is a contract + a composition, not a rewrite: the scripts already make this lossless.

Loop-until-verdict (why no timeout can drop the wait)

The watcher re-arms after every timeout; the .await-cursor makes each re-arm lossless, so a verdict submitted between blocks is drained by the next one. It exits only on a real verdict.

The watcher model is a staffing decision (not "sonnet")

review-loop composes staffing by name to route a watcher task to the floor role — the cheapest reachable model, no capability gate. The literal model is resolved per harness at run time, never baked in.

HarnessFloor / watcher model (resolved by staffing)
Claude Code (this loop)sonnet-5 — the roster floor (never Haiku)
Codex-driven rungpt-5.5 — every role collapses onto it; Claude models unreachable
fallback (floor unreachable)next reachable model per the staffing ladder; never skip the watch

User stories

  • As an orchestrator at an approval gate, I want to hand the watch to a cheap dedicated subagent, so that I neither burn my budget parking on the wait nor abandon it.
  • As a watcher subagent, I want to loop-until-verdict by re-arming review-await.py, so that no single timeout or Bash-tool ceiling drops a long AFK review.
  • As Asher approving in-page hours later, I want my approve / approve_with_nits / request_changes to reliably wake the waiting thread, so that I never have to prod the agent or hand-poll a log (absorbs #23).
  • As a skill author on Codex, I want the watcher's model chosen by staffing for my harness, so that the watch runs on the cheapest reachable model (gpt-5.5), not a hardcoded sonnet the harness can't reach.
  • As the parent thread, I want to resume from the watcher's returned verdict, so that I proceed on approve and revise on request-changes without reading events.jsonl myself.
  • As a maintainer of the PR-merge watch, I want the same delegated-watcher pattern to cover the merge poll, so that both gates share one hardened mechanism.

Implementation decisions

  • The watch is delegated, not held inline. review-loop's await command surface documents that the block on review-await.py is held by a dedicated watcher subagent spawned by the caller, whose sole task is the watch — so it never abandons it. The orchestrator/issue-thread stays free.
  • Model via staffing, never hardcoded. The watch composes the staffing skill by name to route a watcher task to the floor role (cheapest reachable, no capability gate). Resolves to sonnet-5 on Claude Code, gpt-5.5 on Codex, next-reachable on fallback. No model name is baked into review-loop or the scripts.
  • Reuse staffing's existing floor role — no staffing change. "Cheapest reachable, nothing below it" is exactly what a watcher needs; staffing already answers it. review-loop cites the floor/watcher role rather than adding a new role or task-pin (a task-pin is a possible future ergonomic, out of scope here).
  • Loop-until-verdict via re-arm; no script change. The watcher passes --timeout under the Bash ceiling and, on exit 124, re-invokes review-await.py. The existing .await-cursor drains any verdict submitted between blocks, so re-arm is lossless. Investigation confirmed both the --timeout exit and the ~10 min Bash-tool ceiling are defanged by re-arm, so review-await.py/review-server.py stay byte-identical — keeping backlog's bundled copy and every caller non-breaking.
  • Wake = subagent completion. On a real verdict (0/3/10) the watcher returns the verdict as its final message; the Agent-tool result carries it to the parent, which resumes the gate. events.jsonl + .await-cursor remain the durable backstop: a dead watcher loses nothing — a fresh watcher re-drains from the cursor. The parent never polls the log itself (this is the #23 fix).
  • New bundled reference reference/watch.md. Carries the delegated-watch contract in one place, cited by SKILL.md's await command; keeps the existing review-loop.md await-gate section as the low-level exit-code contract and layers the "who holds it / how it wakes" contract on top.
  • Covers both gates. The contract states it applies to the plan/prototype approval gate and the PR-merge watch (the ScheduleWakeup/Monitor poll named in platform.md): both delegate the hold to a floor-staffed watcher that loops-until-signal and wakes the parent.
  • Verification through a path-prefixed mount. The absorbed-#23 acceptance is a runtime submit→await→resume test through the real deployment shape (a stripping proxy mounting the doc under /asher-skills/16/review/), exercising all three verdicts and asserting the parent gets the verdict from the await exit code alone — no events.jsonl read.
  • Staffing of this build-out. Plan authorship + the contract prose (SKILL/watch.md — taste surface) is orchestrator/Claude-builder work (opus-4.8/fable-5); the runtime test harness in evals/ is mechanical (gpt-5.5 via codex exec); probe grading is dual-executor (Claude + codex exec) per probe-evals.md.

Risks
  • A contract-only change relies on callers actually spawning the watcher. Mitigation: the pointer lands at every existing "block on the await script" site (planning.md, platform.md) and the probe grades that an agent routes it correctly, so the contract is discoverable at the point of use.
  • review-loop was a pure root primitive; adding a staffing sibling widens its dependency surface. Mitigation: the sibling is scoped strictly to resolving the watcher model — the serve/annotate/hash core takes no new dependency — and staffing degrades gracefully (missing roster → fallback + report, never a hard stop).
  • The machine sleeping mid-review still stalls a real AFK wait (the existing keep-awake gap in environment.md). Mitigation: out of scope here — the watcher hardens the agent-side drop; the sleep gap is a separate setup choice, noted below.
Out of scope
  • Rewiring backlog's bundled review copy to consume review-loop by name — the separate #6 rewire; this plan keeps backlog non-breaking.
  • Any change to review-await.py/review-server.py behavior or a new --loop flag — the investigation found the cursor already makes re-arm lossless, so no script change is warranted.
  • Adding a named watcher role or task-pin to staffing — the existing floor role answers it; a pin is a future ergonomic.
  • The machine keep-awake gap for reviews that outlast an active run — a docs/agents/environment.md § Presenting setup choice, not this ticket.
