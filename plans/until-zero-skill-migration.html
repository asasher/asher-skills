Until Zero becomes a control-plane skill.

Preserve the financial model, Wallet capture, and calm runway readout. Replace the Lakebed application with workspace-owned state, a dedicated per-installation Runway Capture API, an agent-operated skill, a Shortcuts Playground-authored Wallet Shortcut, and a generated HTML review surface.

Recommendation: retire the app—after a measured cutover. The Lakebed runtime remains the rollback system until real-data parity and a live Wallet cycle both pass.

The problem

Until Zero has a solid engine and a live durable deployment, but its value is trapped inside a bespoke application. The desired operating model is the one already proven in Asher’s workspace: canonical skill source, consumer-owned state and deployment artifacts, capture through a durable queue, and a concise HTML surface for human review.

User stories

  1. S1As Asher, I want each Wallet purchase captured without opening an app, so the runway stays current.
  2. S2As Asher, I want to ask “when do I hit zero?” and receive the expected date, uncertainty range, and the first material driver.
  3. S3As Asher, I want to drop a statement and review a proposed reconciliation before it changes balances, statuses, or recurring rules.
  4. S4As Asher, I want unmapped or ambiguous captures isolated from the trusted runway until I resolve them.
  5. S5As the morning control plane, I want a fresh runway summary and only the decisions that require attention.
  6. S6As a future maintainer, I want to update the skill package without overwriting consumer state or customized deployment files.
  7. S7As Asher, I want the old app available for rollback until the replacement proves parity on my real data and live phone capture.
  8. S8As a new user, I want setup to give me a validated, signed Wallet Shortcut and a short guided iOS automation flow, so I do not have to author the automation actions myself.
  9. S9As a new user, I want setup to deploy my dedicated capture service when Railway is ready, or walk me through the missing Railway steps and resume verification afterward.

Key decisions

  1. D1Build until-zero, not an app-shaped skill. The skill owns domain workflows, deterministic state transitions, projection, and report generation; it does not reproduce every inline-edit interaction.
  2. D2Ship a dedicated Runway Capture API with until-zero. This is not a shared queue and does not alter capture-to-inbox. The skill bundles a small dependency-free server template; setup materializes an editable consumer copy under control-plane/runway/api, following the source/package/instance separation already proven by Capture to Inbox.
  3. D3Separate producer and consumer authority. The Wallet credential can append captures only; a distinct drain credential can list, lease, read, and acknowledge. Query-string tokens are removed, and neither secret is committed.
  4. D10Delegate Shortcut authoring to the external Shortcuts Playground skill. until-zero setup declares the external capability, performs provenance/version/hook disclosure and explicit consent, then asks Shortcuts Playground to build, validate, archive, sign, and verify a minimal “Capture to Runway” Shortcut. The generated Shortcut accepts Transaction input and uses import questions for the dedicated API URL and append-only producer token, so no secret is committed. Setup then guides the user through the Apple-owned step it cannot perform: import the Shortcut, create a personal Transaction automation, select cards, choose Run Immediately, and run the generated Shortcut with the transaction as input.
  5. D11Make Railway the default deployment path, with an effect-verified guided fallback. Setup first materializes the API locally. If the Railway CLI is installed and authenticated, it offers to create or reuse the project, service, persistent volume, and secrets, deploys from the consumer-owned copy, waits for completion, and verifies health plus authentication boundaries. If the CLI or authentication is missing, setup gives exact Railway steps, records the incomplete gate, and resumes at verification after the user finishes. It never fabricates provider state or store secrets in deployment.json.
  6. D4Keep canonical financial state readable and local. Accounts, rules, events, FX, transactions, pending captures, and audit/approval records live in the consumer instance as versioned JSON/JSONL with atomic writes and a single-writer lock.
  7. D5Port the engine to stdlib Python. Skill-repo conventions prohibit a new dependency stack; parity is established against the existing TypeScript oracle before the port becomes authoritative.
  8. D6Mapped captures count; unmapped captures do not. A Wallet charge mapped deterministically to a known account enters as uncleared. Ambiguous captures remain pending and excluded, fixing the current model/comment disagreement.
  9. D7Review changes, not raw state. Reconciliations and material model edits become hash-addressed proposed change sets. The generated HTML shows before/after runway effects; approval is recorded before apply.
  10. D8Integrate softly with the control plane. until-zero is optional. When configured, refresh runs before the morning brief; failures are surfaced without blocking unrelated inbox/task phases.
  11. D9Retire by evidence, not by date. Lakebed is disabled only after frozen-date parity, live Wallet verification, a completed statement workflow, and an agreed rollback window.

Migration sequence

Each phase ends with a red-capable gate; downstream work does not proceed on a hopeful result.

0 · FREEZE & EXPORT

Select the real owner

Secret-safe export of every domain table, IDs, links, exception ranges, order, status, and as-of value. Preserve raw ingest separately under an explicit retention decision.

Gate: row counts reconcile and two settings scopes are explicitly dispositioned.

1 · DEDICATED CAPTURE API

Materialize and deploy

Adapt the proven Capture-to-Inbox template pattern into a runway-only dependency-free Node 20 service with a versioned transaction envelope, append/drain credentials, pagination, a drain lease, acknowledgement, retention, health checks, private file modes, and atomic queue visibility. Setup copies it into the consumer instance with template.json and deployment.json, then provisions Railway automatically when its CLI is authenticated or walks the user through the same gates.

Gate: both automatic and guided setup paths end in a healthy dedicated service whose unauthenticated and producer read attempts fail.

2 · RUNWAY CORE

Port and prove the engine

Materialize the consumer state, import the chosen owner, and run old/new projections on frozen dates, including card-cycle and recurrence edge cases.

Gate: 27 oracle cases plus real-data snapshot parity.

3 · SKILL & REVIEW

Operate through proposals

Implement query, capture refresh, natural-language edits, statement reconcile, proposal/apply, audit log, and self-contained current/snapshot HTML. After the dedicated API passes setup, first-run setup invokes Shortcuts Playground, verifies the signed artifact, records non-secret generation metadata, and walks the user through the personal Transaction automation.

Gate: a fresh executor completes realistic probes and a new user completes API deployment, Shortcut import, and one live capture without authoring actions.

4 · DUAL RUN

Repoint the phone safely

Run the new dedicated Wallet endpoint while Lakebed remains readable. Compare capture, dedupe, account mapping, statement timing, and zero-date output through a real spend/reconcile cycle.

Gate: no missing/duplicate transaction and rollback is rehearsed.

5 · RETIRE & INDEX

Remove the runtime, keep the home

Disable old Shortcut and tokens, archive/export Lakebed, freeze the repo, and retain a thin Project note linking the skill source, instance, deployment, reports, and archive.

Gate: post-retirement morning run and capture smoke test pass.

How it will be checked

Projection seam

Cross-engine golden runner on synthetic edge cases and one sanitized production snapshot, with date fixed and amounts compared in integer minor units.

Capture seam

Dependency-free dedicated-API tests plus both Railway setup branches, live health/auth, Shortcut-shaped post, lease/drain, local commit, ack, replay, and upgrade reconciliation.

Skill seam

Fresh-agent probes for runway query, natural-language rule edit, unmapped transaction, statement proposal, stale approval, and resume after interruption.

Human seam

Offline render and browser screenshots at phone/desktop sizes; keyboard navigation, WCAG contrast, and human approval of the real current report.

Shortcut seam

External dependency consent, Shortcuts Playground validation/signing evidence, non-secret artifact inspection, clean import, personal Transaction automation setup, and one live capture from iPhone.

Deliberately unchecked until the dual-run: prediction accuracy beyond the encoded assumptions. The migration proves semantic equivalence and provenance, not that future cashflow assumptions are objectively correct.

Risks and mitigations
  • Silent financial drift. Freeze dates, retain minor units, compare normalized outputs, and block cutover on any unexplained delta.
  • Wrong-owner migration. Require explicit owner selection and manifest review; never infer from row order or merge scopes.
  • Dedicated-service setup fails halfway. Record each effect-verified gate, reuse healthy Railway resources on resume, never overwrite deployment identity, acknowledge only after local commit, and keep Lakebed capture live until the new endpoint passes a phone smoke test.
  • Second source of truth in HTML. Generate from canonical state and proposals; browser controls annotate/approve only and never directly edit financial records.
  • Over-scoped first release. Defer rich ledger editing, automatic FX, and broad analytics; ship query, capture, proposals, reconciliation, and review first.
  • Sensitive raw history spreads into the workspace. Redact tokens, make raw-log import opt-in, set retention, and keep financial capture outside the general Inbox.
  • The external Shortcut capability is unavailable or changes. Pin and record the reviewed plugin version, stop only the Shortcut setup phase with an exact remediation, preserve already-generated consumer artifacts, and retain the documented manual wiring path as a disclosed fallback rather than silently inventing plist actions.

Out of scope for this migration
  • Rebuilding the full spreadsheet-grid editor in HTML.
  • Bank aggregation, investment tracking, envelopes/budgets, or multi-user collaboration.
  • Changing the pay-in-full card model or redefining certainty bands beyond the explicit unmapped-transaction fix.
  • Automatic server-side FX refresh until capture and migration are proven.
  • Deleting the Git repository or production export; “retire” means disable the runtime and retain a documented, reversible archive.
  • Extracting a generic shared capture-queue skill or modifying the existing Capture-to-Inbox service; Runway capture remains deliberately dedicated.
