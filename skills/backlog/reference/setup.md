# Setup

Prepares a repo for the loop — and reconciles a repo an earlier version already prepared. It inventories any prior installation, scaffolds or upgrades the project playbooks, binds the platforms (tracker, change review, version control, harness) and provisions the tracker's role labels, decides parallelism (preparing worktree isolation when parallel), identifies the seed and check commands, confirms app access, ensures the sibling skills it composes (`staffing`, `review-loop`, `plan`, `prototype`) are present, and verifies the repo is actually ready. Safe to re-run at any time: reconciliation never overwrites repo practice.

## Steps

1. Inventory the installation.
   - List `docs/agents/`: which playbooks exist and which of this skill's `templates/*.md` lack a counterpart. **Match installed playbooks to templates by content, not filename alone** — a template the skill has since renamed (e.g. `pr-reviewer.md` → `change-reviewer.md`) reconciles as a rename of the installed file, carrying every repo-tuned line over verbatim.
   - Legacy artifacts, handled per kind: a `<!-- backlog-templates: ... -->` (or the older `<!-- triage-templates: ... -->`) comment on a playbook's line 1 is a stamp from the retired versioning scheme — delete the line during reconciliation, changing nothing else. `<!-- backlog-section: ... -->` markers in `AGENTS.md`/`CLAUDE.md` (the old picking-models install) are the `staffing` skill's concern — leave them as repo practice and ensure the siblings per steps 7–8 instead.
   - No playbooks → fresh install: scaffold in step 2 and skip reconciliation.
   - Completion criterion: every playbook is classified fresh-scaffold, current, or needs-reconciliation.

2. Scaffold and reconcile the playbooks.
   - Missing → write it from this skill's `templates/`.
   - Needs-reconciliation → **an LLM audit, no version stamps** (the same posture as `staffing` and `setup-asher-skills`): read the installed playbook against the current template, section by section. A section that matches a shipped default — current or older — updates to the current default; a section that diverges is repo practice — leave it, and flag it only where the current template changes the contract around it. Reconciliation preserves repo values (a roster rewrite keeps the repo's model choices, a plan-format change keeps its directory and naming); judge from the text, never from a version token.
   - Never delete repo-authored content. Where a template change and repo practice genuinely conflict, show the conflict and let the user decide.
   - Completion criterion: every template has a reconciled counterpart, and every change or flagged conflict is reported.

3. Bind the platforms.
   - Walk the user through the four ports in `docs/agents/platform.md`, one question each: where issues are tracked (GitHub, a local on-disk tracker, GitLab, other), where changes are reviewed (usually the same host; local otherwise), which version control (git, jj colocated or native), and which harness dispatches threads. Default each answer to what the repo already shows — a `github.com` remote, an existing `.backlog/` directory, a `.jj/` — and confirm rather than interrogate.
   - For a platform with a shipped default (GitHub, local), fill the binding's verb slots from it, adjusted to this repo. For anything else, derive the binding with the user: name the tool or API, exercise every verb live against the real platform, and record only commands that worked; record any verb the platform cannot express as a gap with its fallback, per `platform.md` § Custom bindings.
   - Bind the tracker's **dependency relation** — how `run` learns one issue is blocked by another. GitHub and local have no native blocker field, so the binding is the `- [ ] depends on #N` convention in `backlog-policy.md` § Dependencies. A tracker with a native link type (Jira `blocks`/`is blocked by`, Linear `blocked-by`) binds that relation instead: record, as a verb, how the binding reads an issue's unresolved blockers and how it writes a blocker link, so `run`'s skip-blocked check resolves on this tracker. Where the tracker's live API is not reachable at setup time, ship the relation as a prose contract naming the link type and the intended read/write calls, flagged "verify against the live tracker at first use" — never a fabricated command.
   - Provision the tracker's role labels against **whatever taxonomy already exists** — an inherited tracker may carry dozens of labels, most of them neutral. Inventory every label, then map each readiness role (`ready-for-agent`, `in-flight`, `ready-for-human`, `needs-info`) and work-type role (`bug`, `enhancement`, `refactor`, `draft`) and each exclusion to an existing label where one fits; leave every unrelated label (priority, area, size, component) neutral, untouched. When several existing labels could fill one role, pick one canonical and record the others as aliases the loop treats as that role, rather than renaming the team's labels. Reuse an existing label before creating one — never mint a duplicate of a label that already means the role. Create only the roles no existing label covers, with a clear name and description; creation is an upsert, not an approval gate — cheap and reversible; report what was mapped, aliased, and created. On the local binding there is nothing to provision — roles are frontmatter values; scaffold `.backlog/issues/` instead and offer to import any backlog the user has elsewhere.
   - Write the final mapping (roles, aliases, and the neutral-by-default rule) back into `backlog-policy.md`, the dependency relation into its § Dependencies, and the verified verbs into `platform.md` so the skill's roles resolve to real mechanics.
   - When the tracker's tools are unavailable, list the labels the user must create and the role each fills, and record it as a blocker.
   - Completion criterion: all four bindings are recorded with verified verbs (or explicit gaps), and every role resolves on the tracker — or the missing pieces are listed as an explicit blocker.

4. Choose parallel or sequential, and prepare isolation.
   - Ask the user whether issue work should run in parallel (many worktrees standing up the stack at once) or sequentially (one at a time). Intent and the shared-singleton list set the verdict **asymmetrically**: a preference to serialize is always honored (the user may serialize for any reason), but `parallel-safe` is *never* granted by preference alone — it requires a singleton list with no un-isolated collision. Preference can only tighten toward serialize; only the list can license parallel.
   - **Enumerate the shared singletons first, whatever the intent.** Run `reference/worktree-isolation.md`'s collision probes against the real stack and produce an explicit **shared-singleton list**: every resource two concurrent worktrees would contend for — the database, host ports, `node_modules`/install dirs, build caches, test databases, a cloud deployment or auth tenant — each with its collision mode and whether local namespacing can isolate it. A greenfield repo with no stack yields an empty list honestly; a real app never does. This list, recorded in `environment.md`, is what `run` reads — the one-word verdict is derived from it, not a substitute for it.
   - **Sequential** intent → record `serialize-verification` naming the singletons from the list; no isolation work is needed.
   - **Parallel** intent → cross it with the list and `reference/worktree-isolation.md`: classify the regime and detect any isolation the repo already has.
     - Already isolated (every singleton on the list is namespaced per worktree) → `parallel-safe`; defer to it.
     - Local-isolatable but not isolated → offer to scaffold the isolation layer. Write isolation code only with explicit approval; approved → `parallel-safe`, declined → `serialize-verification`. Until the isolation exists, this is a **hard constraint, not a preference**: with the list's singletons shared, a parallel fan-out would corrupt shared state, so the verdict is `serialize-verification` naming those exact resources even though the user asked for parallel.
     - Cloud-singleton with no per-worktree cloud envs → local isolation can't make it safe; explain why, record `serialize-verification` naming the shared cloud resource, and note the manual path to true parallel (per-worktree deployment + auth tenant).
   - Even when `parallel-safe`, agree with the user which classes of issues must still serialize — destructive operations on a shared tenant, real third-party endpoints without per-worktree credentials, features needing deliberately distinct users — and record them as the **serialized exception lane**.
   - Completion criterion: the parallelism verdict and its exception lane are set, match what the repo can actually support, and any approved isolation scaffolding is in place.

5. Identify seed and checks.
   - Determine how an empty stack is brought to a state that exercises the product's features: a real seed command, a load-from-checked-in-dataset command, or none (state is produced by driving the app). Do not assume a `db:seed` exists — most repos lack one.
   - **Discover the check commands from the repo, then verify each invocation by running it** — do not ask, and do not record a command whose invocation you have not confirmed. Enumerate candidates from the real sources: a machine-readable catalog if present (e.g. `.intent/`), `package.json` scripts, a `Makefile`/`Justfile`/`Taskfile`, `pyproject.toml`/`tox.ini`/`noxfile.py`, `Cargo.toml`, `go.mod` conventions, `build.gradle`/`pom.xml`, and the like. Run each candidate test/lint/typecheck/build command once and judge the **invocation**, not the exit code: record the command verbatim once you have seen it actually launch and execute the check. **A command that runs but reports failures is still recorded** — an inherited repo's suite is often currently red, and a red suite is the repo's real gate in its current state, not a reason to blank the slot; record it with its baseline status (or as a known-red blocker) so `verify` runs it, never omit it. Omit only a command that does not exist or is the wrong invocation (a missing script, a typo, wrong runner). Some checks need the running stack (a suite that hits the DB) and will error until it is up — bring the stack up per step 6 before judging the invocation, and note the dependency. This is what binds `verify` to reality instead of to a guess.
   - **Discover the CI merge-gate** — the exact check set the host runs to gate a merge (e.g. `.github/workflows/*.yml` jobs, GitLab CI stages, the required-status-checks config). Record it as a distinct gate: `verify`'s local checks prove the change, but the PR cannot merge until this CI gate is green (`verifying.md` § CI merge gate, and `change-description.md`'s CI-status line). Where the local commands and the CI jobs diverge, note both.
   - On a re-run, verify every previously recorded command still exists and still runs — a recorded check, seed, CI-gate, or platform-binding command missing from the repo or the machine, or now failing to invoke, is drift; update it.
   - Completion criterion: the seed regime and command (or "none — drive the app"), the check commands (each verified by running), and the CI merge-gate are identified and still real.

6. Confirm app access.
   - Verification and evidence capture only work if the agent can actually exercise the app. Identify the app's form factor(s) and confirm a working **driver** for every surface the loop will verify:
     - **CLI** → shell access and the command entrypoint.
     - **Web** → a browser the agent can drive (e.g. agent-browser, a Playwright MCP, or the harness's built-in browser).
     - **Mobile** → an emulator/simulator the agent can boot, plus a driver that installs the build and interacts with it.
     - **Desktop** → a windowed environment the agent can control (computer-use tooling or an OS automation driver).
   - Confirm an **evidence path** for the same surfaces: screenshots for static states, screen recordings (convertible to GIF) for flows, terminal transcripts for CLI output.
   - Confirm the agent can **mint a session unattended**, matching the app's auth model: a test account, an OTP/magic-link inbox the agent can read, a token in the environment — whatever the model needs.
   - Record the **drive-to-feature path** — beyond booting the stack and seeding, how the agent reaches a state that actually exercises a feature: the entry command or route, the navigation steps, and any preconditions (a logged-in role, a created record, a selected workspace) a criterion needs before it can be observed. Seed (step 5) produces baseline state; this is the path from a running app to the surface under test, and `verify`/`evidence` drive it. Confirm it by walking one feature end to end during the access audit.
   - Where a driver, capture path, auth path, or drive-to-feature path is missing, help set it up — install the tool, create the test account, wire the inbox — with the user's approval. Anything that cannot be set up is a blocker for the surfaces it covers, and `verify`/`evidence` will inherit it.
   - Completion criterion: every app surface the loop will verify has a named driver, capture method, auth path, and a walked drive-to-feature path — or an explicit blocker.

7. Ensure the review-loop skill (presentation & review).
   - The presentation surface and the interactive review loop are owned by the **`review-loop`** skill (composed by name). Ensure it is present: the `setup-asher-skills` installer guarantees the siblings; absent it, tell the user to install `review-loop`.
   - review-loop's own setup records this repo's surface config (tailnet root, surface dir, publish/proxy commands, keep-awake) into `environment.md` § Presenting. backlog does not install the surface itself.
   - Degrade note: absent `review-loop`, presentation falls back to a local open of the rendered document.
   - Completion criterion: `review-loop` is present and its surface config resolves, or the gap is reported.

8. Ensure the staffing skill (model roster).
   - The model roster (rankings, roles, capability matrix, task-pins, fallback ladder) is owned by the **`staffing`** skill (composed by name). Ensure it is present; staffing's own setup writes the roster — global base plus this project's delta — with user consent, probing what this machine can actually staff. backlog does not install the picking-models sections.
   - The enhancement and diagnose branches also compose the **`plan`** and **`prototype`** siblings by name — ensure those are present too; each installs its own `docs/agents/` playbook (planning, prototyping).
   - Completion criterion: the four siblings (`staffing`, `review-loop`, `plan`, `prototype`) resolve, or the gaps are reported.

9. Write the contracts into the playbooks.
   - Record the results in `docs/agents/environment.md`: dev-stack startup, isolation regime and how to bring up an isolated stack, the **enumerated shared-singleton list** from step 4, seed regime and command, the **drive-to-feature path** from step 6, auth/session minting, the app drivers and evidence-capture tooling from the access audit, and the parallelism verdict with the singletons that force it plus its serialized exception lane. `environment.md` § Presenting is a pointer the `review-loop` skill fills (its setup records the surface config); § Model staffing is the `staffing` skill's roster (its setup writes it) — backlog does not write either here. The platform bindings live in `platform.md` from step 3. Record the **verified check commands and the CI merge-gate** from step 5 in `verifying.md`, and the repo's evidence expectations in `evidence.md` — for a real running app that means real check/test output plus screenshots driven through the app, not agent-authored probes; the presentation contract for the bound review surface lives there too.
   - Offer to fill the remaining baselines — the dependency relation and neutral-label rule in `backlog-policy.md`, branch conventions, plan format, review criteria — with this repo's real values. The templates are baselines, not the contract.
   - Completion criterion: `environment.md` states the isolation, seed, and parallelism contracts (its § Presenting and § Model staffing being sibling-owned pointers) and `platform.md` the bindings the loop reads, or the user has chosen to keep a baseline section.

10. Fold in existing repo practice.
   - The scaffolded playbooks are self-contained: the debugging discipline, test-first build technique, and review standards ship inlined, so nothing external is required. But a repo may already have its own practice — a dev-loop skill, a testing guide, debugging docs, review checklists.
   - Where one exists, offer to fold it into (or substitute it for) the matching playbook section, so the loop follows house practice instead of the shipped default. Edit only with the user's approval.
   - Completion criterion: each playbook either keeps its inlined default or records the repo's own practice, per the user's choice.

11. Verify readiness and confirm.
   - Offer a smoke test of the recorded contracts — bring up the stack, seed it, mint a session, drive one interaction through the recorded driver, capture one throwaway artifact, round-trip the tracker binding (create a throwaway issue, label it, comment, close it — or the local-file equivalent), verify an artifact renders inline on the bound review surface per `evidence.md`'s presentation contract, publish a throwaway page to the presentation surface and confirm it resolves, and run the check gate once — so a wrong command or a broken binding surfaces now, not mid-loop. When the verdict is `parallel-safe`, run it in a throwaway worktree to confirm isolation actually holds; tear the worktree down after. On a re-run, also sweep orphaned surface path handlers whose worktrees are gone, and sweep the review hub registry via the **`review-loop`** skill's sweep.
   - Report a readiness checklist: playbooks present and reconciled; platform bindings verified per verb; tracker roles provisioned; dependency convention set; seed and checks identified and real; app driver, evidence capture, and auth confirmed per surface (and smoke-tested if run); the four sibling skills (`staffing`, `review-loop`, `plan`, `prototype`) present and their playbooks/surface-config resolving; parallelism verdict and exception lane; playbooks tailored to house practice where the repo has one; migrations applied and conflicts resolved.
   - Completion criterion: the user has a clear ready / not-ready verdict for each item and knows whether `backlog` will verify in parallel or serialized.
