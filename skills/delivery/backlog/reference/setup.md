# Setup

Prepares a repo for the loop — and reconciles a repo an earlier version already prepared. It inventories any prior installation, scaffolds or upgrades the project playbooks, binds the platforms (tracker, change review, version control, harness) and provisions the tracker's role labels, decides parallelism (preparing worktree isolation when parallel), identifies the seed and check commands, confirms verification-surface access, ensures its declared sibling closure is present, and verifies the repo is actually ready. Safe to re-run at any time: reconciliation never overwrites repo practice.

## Steps

1. Inventory the installation.
   - List `docs/agents/` and determine this repo's work domain from the installed `backlog-policy.md` § Work domain; if the section is absent, use `software` (a legacy install); a fresh install has no playbook to read — the domain is asked in step 2. Resolve the **scaffold set** for that domain in two passes: overlay `templates/common/` with `templates/<domain>/`, with a same-name pack file shadowing common; then, for each required backlog-owned step playbook absent from that overlay, add the same-name `templates/software/` baseline as a flagged stand-in. The required backlog-owned step names are `implementing`, `verifying`, `evidence`, `refactoring`, `change-reviewer`, and `change-fixer`; diagnosis is reconciled separately by the `diagnosing-bugs` sibling. Compare the installed playbooks against that resolved scaffold set and identify which playbooks lack a counterpart. **Match installed playbooks to templates by content, not filename alone** — a template the skill has since renamed (e.g. `pr-reviewer.md` → `change-reviewer.md`) reconciles as a rename of the installed file, carrying every repo-tuned line over verbatim.
   - Legacy artifacts, handled per kind: a `<!-- backlog-templates: ... -->` (or the older `<!-- triage-templates: ... -->`) comment on a playbook's line 1 is a stamp from the retired versioning scheme — delete the line during reconciliation, changing nothing else. `<!-- backlog-section: ... -->` markers in `AGENTS.md`/`CLAUDE.md` (the old picking-models install) are the `staffing` skill's concern — leave them as repo practice and ensure the siblings per steps 7–8 instead.
   - No playbooks → fresh install: scaffold in step 2 and skip reconciliation.
   - Completion criterion: the work domain is known (recorded or defaulted; on a fresh install, deferred to step 2's question), and every playbook in the resolved scaffold set is classified fresh-scaffold, current, or needs-reconciliation. For `skill-authoring`, backlog owns ten counterparts: three unshadowed common files, three native pack files, and four flagged software stand-ins; `diagnosing-bugs setup` owns the eleventh project playbook.

2. Choose the work domain; scaffold and reconcile the playbooks.
   - Fresh install: **before scaffolding anything**, ask the user "what kind of work does this repo track?" with exactly six options: `software` / `skill-authoring` / `writing` / `research` / `ops` / `general`. Default the answer from repo evidence and confirm rather than interrogate, in the same posture as step 3's binding questions: a skill-authoring repo (`SKILL.md` files under `skills/<name>/`) defaults to `skill-authoring`; a code repo (`package.json`, `src/`, a test runner) defaults to `software`; a manuscript defaults to `writing`; ambiguous evidence defaults to `software`. Record the answer in the installed `backlog-policy.md` § Work domain.
   - Re-run: the recorded domain stands. If § Work domain is absent, use `software` and record it there during reconciliation. Changing an installed repo's domain is a user decision, never a silent reconciliation outcome.
   - Scaffolding and reconciliation draw from the **resolved scaffold set**: first overlay `templates/common/` with `templates/<domain>/`, with a same-name pack file shadowing common; then add the same-name `templates/software/` baseline as a flagged stand-in for every required backlog-owned step playbook absent from that overlay. The required backlog-owned step names are `implementing`, `verifying`, `evidence`, `refactoring`, `change-reviewer`, and `change-fixer`; diagnosis is sibling-owned.
   - Missing-baseline degrade: when the chosen domain has no shipped pack (today, every domain except `software` and `skill-authoring`), install the `templates/software/` step baselines as stand-ins. When a shipped pack omits a required step baseline, install only each missing same-name `templates/software/` baseline as a stand-in. Add one flag line to every installed stand-in's header naming it a code-flavored baseline to tailor for `<domain>`, and name every gap in the setup report. Never leave a required playbook uninstalled. For `skill-authoring`, its pack `environment.md` shadows common and only `implementing.md`, `refactoring.md`, `change-reviewer.md`, and `change-fixer.md` come from software as flagged stand-ins.
   - Missing → write it from the resolved scaffold set, including any required flagged stand-in.
   - Needs-reconciliation → **an LLM audit, no version stamps** (the same posture as `staffing` and `setup-asher-skills`): read the installed playbook against the current template, section by section. A section that matches a shipped default — current or older — updates to the current default; a section that diverges is repo practice — leave it, and flag it only where the current template changes the contract around it. Reconciliation preserves repo values (a roster rewrite keeps the repo's model choices, a plan-format change keeps its directory and naming); judge from the text, never from a version token.
   - Never delete repo-authored content. Where a template change and repo practice genuinely conflict, show the conflict and let the user decide.
   - Completion criterion: the domain is chosen and recorded, every playbook in the resolved scaffold set has a reconciled counterpart, and every change, degraded stand-in, and flagged conflict is reported. For `skill-authoring`, require all ten backlog-owned counterparts—three unshadowed common files, three native pack files, and four flagged software stand-ins—plus the sibling-owned diagnosis playbook.

3. Bind the platforms.
   - Walk the user through the four ports in `docs/agents/platform.md`, one question each: where issues are tracked (GitHub, a local on-disk tracker, GitLab, other), where changes are reviewed (usually the same host; local otherwise), which version control (git, jj colocated or native), and which harness dispatches threads. Default each answer to what the repo already shows — a `github.com` remote, an existing `.backlog/` directory, a `.jj/` — and confirm rather than interrogate.
   - For a platform with a shipped default (GitHub, local), fill the binding's verb slots from it, adjusted to this repo. For anything else, derive the binding with the user: name the tool or API, exercise every verb live against the real platform, and record only commands that worked; record any verb the platform cannot express as a gap with its fallback, per `platform.md` § Custom bindings.
   - For each sibling-harness dispatch, bind and smoke-test the `staffing` contract: a watched native wrapper
     named for the external model/task, staffed by the cheapest native model allowed by the floor, supervising
     a bounded non-interactive command and returning raw output plus lifecycle status without synthesis. The
     parent verifies the effect. Record the spawn request or child metadata that proves wrapper staffing; if
     the harness cannot select or report it, keep the route's observability result but mark floor/cost
     compliance unproven.
   - Discover the tracker's **native dependency relation** before choosing a fallback, then exercise and record both read and write verbs. GitHub reads `.issue_dependencies_summary` and writes `dependencies/blocked_by` using the blocker's numeric issue id; local uses `deps` frontmatter. For another tracker, probe its native link against setup-owned throwaways and clean them up. If native edges are absent or cannot be exercised, record the explicit fallback and reason instead of an intended command; `run` uses only the recorded path.
   - Provision the tracker's role labels against **whatever taxonomy already exists** — an inherited tracker may carry dozens of labels, most of them neutral. Inventory every label, then map each readiness role (`ready-for-agent`, `in-flight`, `ready-for-human`, `needs-info`) and work-type role (`bug`, `enhancement`, `refactor`, `draft`) and each exclusion to an existing label where one fits; leave every unrelated label (priority, area, size, component) neutral, untouched. When several existing labels could fill one role, pick one canonical and record the others as aliases the loop treats as that role, rather than renaming the team's labels. Reuse an existing label before creating one — never mint a duplicate of a label that already means the role. Create only the roles no existing label covers, with a clear name and description; creation is an upsert, not an approval gate — cheap and reversible; report what was mapped, aliased, and created. On the local binding there is nothing to provision — roles are frontmatter values; scaffold `.backlog/issues/` instead and offer to import any backlog the user has elsewhere.
   - Write the final mapping (roles, aliases, and the neutral-by-default rule) back into `backlog-policy.md`, the dependency relation into its § Dependencies, and the verified verbs into `platform.md` so the skill's roles resolve to real mechanics.
   - When the tracker's tools are unavailable, list the labels the user must create and the role each fills, and record it as a blocker.
   - Completion criterion: all four bindings are recorded with verified verbs (or explicit gaps), dependency read/write is exercised or has an explicit fallback, and every role resolves on the tracker — or the missing pieces are listed as an explicit blocker.

4. Choose parallel or sequential, and prepare isolation.
   - Ask the user whether issue work should run in parallel (many worktrees standing up the stack at once) or sequentially (one at a time). Intent and the shared-singleton list set the verdict **asymmetrically**: a preference to serialize is always honored (the user may serialize for any reason), but `parallel-safe` is *never* granted by preference alone — it requires a singleton list with no un-isolated collision. Preference can only tighten toward serialize; only the list can license parallel.
   - **Enumerate the shared singletons first, whatever the intent.** Run `reference/worktree-isolation.md`'s collision probes against the real stack and produce an explicit **shared-singleton list**: every resource two concurrent worktrees would contend for — the database, host ports, `node_modules`/install dirs, build caches, test databases, a cloud deployment or auth tenant — each with its collision mode and whether local namespacing can isolate it. A greenfield repo with no stack yields an empty list honestly; a real app never does. This list, recorded in `environment.md`, is what `run` reads — the one-word verdict is derived from it, not a substitute for it.
   - **Sequential** intent → record `serialize-verification` naming the singletons from the list; no isolation work is needed.
   - **Parallel** intent → cross it with the list and `reference/worktree-isolation.md`: classify the regime and detect any isolation the repo already has.
     - Already isolated (every singleton on the list is namespaced per worktree) → `parallel-safe`; defer to it.
     - Local-isolatable but not isolated → offer to scaffold the isolation layer. Write isolation code only with explicit approval; approved → `parallel-safe`, declined → `serialize-verification`. Until the isolation exists, this is a **hard constraint, not a preference**: with the list's singletons shared, a parallel fan-out would corrupt shared state, so the verdict is `serialize-verification` naming those exact resources even though the user asked for parallel.
     - Cloud-singleton with no per-worktree cloud envs → local isolation can't make it safe; explain why, record `serialize-verification` naming the shared cloud resource, and note the manual path to true parallel (per-worktree deployment + auth tenant).
   - Even when `parallel-safe`, agree with the user which classes of issues must still serialize — destructive operations on a shared tenant, real third-party endpoints without per-worktree credentials, features needing deliberately distinct users — and record them as the **serialized exception lane**.
   - Completion criterion: the parallelism verdict and its exception lane are set, match what the repo can actually support, and any approved isolation scaffolding is in place.

5. Identify seed and checks.
   - Determine how an empty verification surface is brought to a state that exercises the tracked work: a skill's checked-in `evals/` scenarios and prewritten answer key, a real seed command, a load-from-checked-in dataset, or none (state is produced while exercising the surface). Do not assume a `db:seed` exists — most repos lack one. For `skill-authoring`, record the checked-in probe scenarios as seed state and do not invent an app or stack.
   - **Discover the check commands from the repo, then verify each invocation by running it** — do not ask, and do not record a command whose invocation you have not confirmed. Enumerate candidates from the real sources: a machine-readable catalog if present (e.g. `.intent/`), `package.json` scripts, a `Makefile`/`Justfile`/`Taskfile`, `pyproject.toml`/`tox.ini`/`noxfile.py`, `Cargo.toml`, `go.mod` conventions, `build.gradle`/`pom.xml`, and the like. Run each candidate test/lint/typecheck/build command once and judge the **invocation**, not the exit code: record the command verbatim once you have seen it actually launch and execute the check. **A command that runs but reports failures is still recorded** — an inherited repo's suite is often currently red, and a red suite is the repo's real gate in its current state, not a reason to blank the slot; record it with its baseline status (or as a known-red blocker) so `verify` runs it, never omit it. Omit only a command that does not exist or is the wrong invocation (a missing script, a typo, wrong runner). Some checks need the running stack (a suite that hits the DB) and will error until it is up — bring the stack up per step 6 before judging the invocation, and note the dependency. This is what binds `verify` to reality instead of to a guess.
   - **Discover the CI merge-gate** — the exact check set the host runs to gate a merge (e.g. `.github/workflows/*.yml` jobs, GitLab CI stages, the required-status-checks config). Record it as a distinct gate: `verify`'s local checks prove the change, but the PR cannot merge until this CI gate is green (`verifying.md` § CI merge gate, and `change-description.md`'s CI-status line). Where the local commands and the CI jobs diverge, note both.
   - On a re-run, verify every previously recorded command still exists and still runs — a recorded check, seed, CI-gate, or platform-binding command missing from the repo or the machine, or now failing to invoke, is drift; update it.
   - Completion criterion: the seed regime and command (or checked-in scenario / "none — state is produced while exercising the surface"), the check commands (each verified by running), and the CI merge-gate are identified and still real.

6. Confirm verification-surface access.
   - Verification and evidence capture only work if the agent can exercise the changed behavior. Identify its form factor(s) and confirm a working **driver** for every surface the loop will verify:
     - **Executor-harness skill behavior** → a fresh in-session executor plus an independent CLI executor, each able to load the skill context and answer a checked-in situated probe.
     - **CLI** → shell access and the command entrypoint.
     - **Web** → a browser the agent can drive (e.g. agent-browser, a Playwright MCP, or the harness's built-in browser).
     - **Mobile** → an emulator/simulator the agent can boot, plus a driver that installs the build and interacts with it.
     - **Desktop** → a windowed environment the agent can control (computer-use tooling or an OS automation driver).
   - Confirm an **evidence path** for the same surfaces: cited executor transcripts plus a verdict table for skill behavior, terminal transcripts for CLI output, screenshots for static visual states, and screen recordings (convertible to GIF) for flows.
   - Confirm the agent can **mint any required session unattended**, matching the surface's auth model: a fresh executor session, test account, OTP/magic-link inbox, or token in the environment — whatever the driver needs.
   - Inventory verification data now, not during verify: standing accounts/tenants and permissions; largest feasible real scale; any plan-approved synthetic substitute and the criterion classes where it is valid; collision-safe per-issue fixture names; fixture retention through final evidence; and the named owner/command for cleanup. Record absent affordances as blockers rather than inventing accounts, scale, or substitutes.
   - Record the **exercise-to-criterion path** — after readying and seeding the surface, how the agent reaches an observable criterion: load a skill and submit the situated scenario; invoke the CLI entrypoint; or follow the app route/navigation and preconditions. Seed (step 5) produces baseline state; this is the path `verify` and `evidence` drive. Confirm one path end to end during the access audit.
   - Where a driver, capture path, auth path, or exercise-to-criterion path is missing, help set it up — install the tool, create the test account, wire the inbox — with the user's approval. Anything that cannot be set up is a blocker for the surfaces it covers, and `verify`/`evidence` will inherit it.
   - Completion criterion: every verification surface has a named driver, capture method, any required auth path, a walked exercise-to-criterion path, and a complete verification-data inventory — or an explicit blocker.

7. Ensure the review-loop skill (presentation & review).
   - The presentation surface and the interactive review loop are owned by the **`review-loop`** skill (composed by name). Ensure it is present: the `setup-asher-skills` installer guarantees the siblings; absent it, tell the user to install `review-loop`.
   - review-loop's own setup records this repo's surface config (tailnet root, surface dir, publish/proxy commands, keep-awake) into `environment.md` § Presenting. backlog does not install the surface itself.
   - Degrade note: absent `review-loop`, presentation falls back to a local open of the rendered document.
   - Completion criterion: `review-loop` is present and its surface config resolves, or the gap is reported.

8. Ensure the staffing skill (model roster).
   - The model roster (rankings, roles, capability-provider registry, pins, fallback ladder) is owned by the **`staffing`** skill (composed by name). Ensure it is present; staffing's own setup writes the roster — global base plus this project's delta — with user consent, probing what this machine can actually staff. backlog does not install the picking-models sections.
   - The enhancement and diagnose branches also compose **`plan`**, **`prototype`**, and **`diagnosing-bugs`** by name. Ensure they are present; invoke `diagnosing-bugs setup` to reconcile its owned playbook, while plan and prototype may use their bundled defaults.
   - Completion criterion: the declared siblings (`diagnosing-bugs`, `staffing`, `review-loop`, `plan`, `prototype`) resolve, or the gaps are reported.

9. Write the contracts into the playbooks.
   - Record the results in `docs/agents/environment.md`: verification-surface startup/readiness, isolation regime and how to ready an isolated surface, the **enumerated shared-singleton list** from step 4, seed regime and command, the **exercise-to-criterion path** from step 6, auth/session minting, drivers and evidence-capture tooling from the access audit, and the parallelism verdict with the singletons that force it plus its serialized exception lane. `environment.md` § Presenting is a pointer the `review-loop` skill fills (its setup records the surface config); § Model staffing is the `staffing` skill's roster (its setup writes it) — backlog does not write either here. The platform bindings live in `platform.md` from step 3. Record the **verified check commands and the CI merge-gate** from step 5 in `verifying.md`, and the repo's evidence expectations in `evidence.md` — skill behavior uses cited executor transcripts and verdict tables; a real running app uses real check/test output plus screenshots driven through the app, not agent-authored probes. The presentation contract for the bound review surface lives there too.
   - Offer to fill the remaining baselines — the dependency relation and neutral-label rule in `backlog-policy.md`, branch conventions, plan format, review criteria — with this repo's real values. The templates are baselines, not the contract.
   - Completion criterion: `environment.md` states the isolation, seed, and parallelism contracts (its § Presenting and § Model staffing being sibling-owned pointers) and `platform.md` the bindings the loop reads, or the user has chosen to keep a baseline section.

10. Fold in existing repo practice.
   - The scaffolded playbooks are self-contained for backlog-owned techniques; debugging is supplied by the `diagnosing-bugs` sibling plus its repo delta. A repo may already have its own practice — a dev-loop skill, testing guide, debugging docs, or review checklist.
   - Where one exists, offer to fold it into (or substitute it for) the matching playbook section, so the loop follows house practice instead of the shipped default. Edit only with the user's approval.
   - Completion criterion: each playbook either keeps its inlined default or records the repo's own practice, per the user's choice.

11. Verify readiness and confirm.
   - Offer a smoke test of the recorded contracts — ready the verification surface, seed it, mint any required session, exercise one criterion through the recorded driver, capture one throwaway proof artifact, round-trip the tracker binding (create two throwaway issues, write/read their dependency edge, then label, comment, and close them—or the local-file equivalent), verify an artifact renders inline on the bound review surface per `evidence.md`'s presentation contract, publish a throwaway page to the presentation surface and confirm it resolves, and run the check gate once — so a wrong command or a broken binding surfaces now, not mid-loop. When the verdict is `parallel-safe`, run it in a throwaway worktree to confirm isolation actually holds; tear the worktree down after. On a re-run, also sweep orphaned surface path handlers whose worktrees are gone, and sweep the review hub registry via the **`review-loop`** skill's sweep.
   - Report a readiness checklist: playbooks present and reconciled; platform bindings verified per verb; tracker roles provisioned; dependency read/write exercised or explicit fallback recorded; seed and checks identified and real; verification-surface driver, evidence capture, and any required auth confirmed per surface (and smoke-tested if run); the declared sibling closure present and owner setup branches reconciled; parallelism verdict and exception lane; playbooks tailored to house practice where the repo has one; migrations applied and conflicts resolved.
   - Completion criterion: the user has a clear ready / not-ready verdict for each item and knows whether `backlog` will verify in parallel or serialized.
