Good Work — how the user and the agent find good work together

One person, one agent, one workspace. The agent defines what good work means for this person, then runs the campaign operation — targets, reach-outs, applications, inbound mail — while the user holds every gate that touches the outside world. Everything either of them produces lands as a durable artifact in the goodwork/ workspace.

3Sequence — one outbound reach-out or application

The execution contract: nothing reaches the world without a truth-checked draft, a hash-matched approval, and a recorded proof. The server can request; only the agent writes; only the user sends.

Agent Drafts the message or application package from the profile and niche story.
truth gate — nothing invented, ever
Agent Stores the artifact, computes its content hash, and presents the exact final text for review.
User Reads the real text — on the phone approval page, the kanban, or in chat — and clicks approve (or edits).
Local server Appends the click as a request event to events.jsonl — a request, not yet an approval.
Agent Validates the event and writes the approval record to approvals.jsonl.
hash gate — text unchanged since seen evidence gate — ~70% of must-haves evidenced quota gate — weekly cap is hard
Agent Executes on the best available rung of the ladder:
① Gmail draft — placed in the user's mailbox  ② ATS-direct (Greenhouse/Lever/Ashby)
③ authenticated Chrome for logged-in boards  ④ manual package — draft + field-by-field instructions.
User On rung ① presses Send in Gmail — the physical gate. On rung ④ performs the action themselves.
Agent Records the outcome on the pipeline card and pre-decides the follow-up so anxiety doesn't improvise: +3 business days → +7 days → monthly nurture, never more.
proof gate — screenshot / confirmation ref required

4Sequence — an inbound message (reconcile)

Runs before daily, because inbound mail decides today's queue. The sweep is strictly read-only — nothing is sent from any channel while sweeping.

Agent Sweeps in order: Gmail via connector → LinkedIn via the persistent Chrome profile → WhatsApp read-only, only if the user explicitly accepted that risk at setup.
Agent Matches each item to a pipeline card by thread ID, sender/domain, company, role, URLs, quoted text.
Agent Confident match → advances the stage (reply · booked · referral · screen · interview · offer · rejection · ghost) and records a reason code. Every open card keeps a next action + due date.
User Low-confidence match → lands in unmatched_replies; the user says which thread it belongs to before any stage changes.
Agent Drafts responses as Gmail drafts; interview requests become calendar-slot proposals (holds are created only after approval).
User Sends the reply drafts, confirms the slot that works.
Agent Tags what the market is teaching into evidence-inbox.json — repeated rejections for a missing skill, roles the user keeps dismissing, calls that energized them — and refreshes today's priorities.

5Everything that gets created

The five artifact families, each with its home in the workspace. JSON when a matcher or approval flow consumes it; Markdown when a person does.

🧾 Evidence

from interviews, prototypes, and what the market keeps saying
  • Profile claims marked reported / evidenced / tested in PROFILE.md, with a dated changelog
  • evidence-inbox.json — queued signals: rejection patterns, dismissed-role patterns, energizing calls
  • Good Time Journal lines in JOURNAL.md — did · win · energy · flow, instrumentation not diary
  • Prototype & craft debriefs in EXPERIMENTS.md
  • Reason codes on every rejection/retirement — profile evidence, not just bookkeeping

📄 Documents

from assets + niche, mechanics delegated to eloquent when present
  • CV/resume plus truthfully tailored variants and cover letters
  • LinkedIn headline & summary telling the bridge story — same story on every surface
  • Portfolio case studies and the niche two-pager in NICHE.md
  • Proof-of-work artifacts — the repo, teardown, or essay an insider actually inspects, one per 1–2 weeks
  • A consistency audit: every surface names the same niche, strengths, and proof links

🤝 Captured reach-outs

from outreach — the human layer that carries the search
  • One pipeline.json card per relationship thread, with thread IDs and warmth
  • Draft records with content hashes — what was said, provably the approved version
  • Per-conversation prep sheets: their background, why-them, ten TIARA questions
  • Call debriefs feeding niche hypotheses back into NICHE.md
  • The 3-day / 7-day / nurture follow-up schedule, pre-decided per thread

📬 Applications

from apply — deliberate, evidence-gated, quota-capped
  • The go/no-go score: fit, evidence coverage, warmth, posting freshness
  • The truthful application package — keywords mirrored only where evidence exists
  • An approval record in approvals.jsonl bound to the exact submitted text
  • Submission proof — confirmation screenshot or reference on the card
  • Weekly funnel numbers in metrics.json: sent → screens → interviews → offers

💬 Messages

from reconcile — every inbound touch, matched and kept small
  • Reply records nested on their pipeline cards, with stage signals
  • The unmatched_replies queue awaiting the user's judgment
  • Gmail reply drafts waiting on the user's Send
  • Calendar-slot proposals for interview requests
  • A reply_digest per card — old threads compacted, stage-changing IDs kept

6Why this is safe to run on your career

Three actors, three strictly separated powers.

🤖 The agent is the sole writer

Every state file is written by the agent only — cumulatively, never regenerated. It validates every request before acting and may never invent experience, metrics, credentials, or relationships.

🖥 The server can only ask

The local UI (kanban, phone approval page) appends request events to events.jsonl and nothing else. A click is a request; the agent turns it into an approval only after the hash check.

🧑 The user holds the physical gates

Approvals bind to the exact text seen. Email leaves only when the user presses Send in Gmail. Quotas are hard caps. Every submission needs proof. The definition of good work itself is elicited, never assumed.
