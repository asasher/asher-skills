Proposed shape, in one breath.
  • You route; skills are legible. There is no router stage. Each skill states plainly what it does and what artifact it leaves behind, so invoking the right one is obvious. The one automated nudge lives in grooming: "this is small enough to implement directly" vs "this needs a spec first" — moved out of the run loop (#80), stamped before execution.

No new skills: groom = organize, run = schedule, issue loop = one issue → reviewed PR, and run already takes a queue of one. Over-decomposition is as bad as under-decomposition.
  • Impeccable fits as a lane, not a spine. That was the real clash. It's invoked at three named points (project setup, spec's design direction, verification) and never owns the workflow. Kept external; mechanics absorbed.

2 · Fidelity, not reproducibility

Fair challenge. Dropping the word: nobody needs to-spec to be a deterministic function that re-derives the same spec twice. The spec is written once, in the same context window as the conversation that produced it. What actually went wrong in practice (your words, turn one: outputs you couldn't get to come out well "in a reliable fashion") is a fidelity problem — material things agreed in conversation silently missing from the artifact, or things never asked at all. Two honest observations survive the reframe:

  • "Go back to the context window" has a shelf life. Sessions end, context compacts. Whatever should outlive the conversation has to be written somewhere during it. That's all the extraction discipline is: domain-modeling's "update CONTEXT.md right there, don't batch it up" — terms → glossary, hard-to-reverse decisions → one-paragraph ADRs (3-gate test: hard-to-reverse ∧ surprising ∧ real trade-off), everything else deliberately evaporates. No transcript, no ledger.
  • Fidelity is checkable at publish time. Before the spec or tickets go out: does every material decision from the conversation appear, and is every open thread classified — delegated (executor's call), deferred (parked, with a home), or blocking (can't publish)? A cheap audit pass, not a determinism guarantee.

"Pure synthesis," in plain words (it was jargon, sorry): to-spec asks no new questions and invents no new decisions — it only writes down what the conversation already settled. Matt's rule ("Do NOT interview the user — just synthesize what you already know") keeps the interview and the writing honest about which mode they're in. The "crystallized artifacts" part just meant: since the interview wrote terms and decisions down as it went, the spec-writer draws on those notes plus the conversation, rather than on vibes and memory alone.

4 · Where things live

Spec: HTML, in the repo, reviewed like plans are today. Written by to-spec as a rendered document (decisions, seams, UX direction, out-of-scope), served through review-loop for annotation and sign-off, committed on approval. It's the same presentation discipline this repo already uses for plans — the spec simply takes over the slot.

Tracker projection stays thin. A tracking issue carries the gist, a link to the canonical spec, and the slice list wired with native blocked-by edges so the frontier renders in GitHub's own UI. Links and state, never content — one source of truth, no drift.

The to-tickets disk bug stays fixed the same way: tickets always publish through platform.md's verified tracker verbs; local files only when the binding says local.

The dev tail is the same in every shape — only its inputs change

Your question about verification and evidence across the entryways has a clean answer once the tail is seen as part of the issue loop, not part of the queue: verify → adversarial review → evidence attach to the PR boundary, so every shape that produces a code change flows through them. What varies per entryway is what feeds each gate:

This is also the argument for the seam-naming below: because the gates ride the issue loop, running that loop with or without a queue around it loses nothing.

Backlog's seams — name them, don't split them

Correction from chat, and it simplifies things: the per-issue orchestrator already exists and already has a name — backlog's issue loop (reference/issue-loop.md), which coordinates the sub-steps: implement.md (the build), verify.md, adversarial-review.md, evidence.md, with diagnose.md/refactor.md as work-type branches. Rev 2 accidentally reused "implement" for the loop itself — wrong altitude; "implement" stays the build sub-step's name.

And with that correction, the decomposition mostly evaporates as a skill-splitting project — over-decomposition is as bad as under-decomposition, and backlog should stay the thin head. What remains is naming its three internal seams as contracts, all inside the one skill:

  • groom = organize: the tracker as truth — admission audit, route stamps, serialized tracker writes, verify-first triage discipline.
  • run = schedule: dependency waves, parallelism, worktrees, dispatch, liveness — and it already supports a single issue (a queue of one), so interactive chat-and-build needs no new skill: it's the same issue loop, invoked without waves.
  • issue loop = one issue → reviewed PR, carrying the dev tail above.

The only structural change anyone actually needs is #80's: the planning gate leaves the issue loop's enhancement branch (replaced by groom's route stamp + the spec stage upstream). Everything else is documentation of seams that already exist.

6 · The interview: batch-grill-me × domain-modeling

Adopted as-is in spirit — the composition Matt gestures at in 7 lines (grill-with-docs), with his batch shape rather than one-at-a-time:

  • Frontier rounds. Map decisions as a tree; each round asks every question whose prerequisites are settled; answers reshape the tree and unblock the next round. Facts get sub-agents (a running lookup blocks only its downstream questions). Recommended answer attached to each question.
  • Extraction inline. domain-modeling alongside: challenge terms against the glossary, sharpen fuzzy language, stress-test with concrete scenarios, cross-reference claims against the code — and write CONTEXT.md/ADRs the moment something lands.
  • Done when the frontier is empty and you confirm shared understanding — then to-spec (or straight to implement) in the same window.

"Bounded rounds," explained — and then mostly withdrawn. It meant: cap the number of questions per round (say 4) so a round never lands as a 15-question wall. Since the full-frontier batch is precisely what you like, the default is Matt's: ask the whole frontier. Early frontiers are naturally small (few prerequisites settled), so the wall problem mostly self-limits; keep a common-sense escape valve — if a frontier balloons, split it by dependency cluster and say so. Not a numeric rule, just manners.

Two refinements kept from Impeccable's interview craft, both compatible with full-frontier: one topic per question (no compound either-ors), and assert-then-confirm when the evidence already points one way ("This reads as X — confirm?" beats a four-option menu).

7 · Impeccable — the fit, settled

The real issue was never Impeccable's quality — it was that it ships a workflow (init → shape → craft → critique/audit → polish) that competes with ours. Resolution: it plugs in at three named points and never owns the spine:

Lifecycle pointWhat Impeccable does thereOurs vs theirs
Project setupinit/document: PRODUCT.md + DESIGN.md written once, read by everything afterThe artifacts become ours (repo-owned, listed in § 2's map); Impeccable is one writer of them
Interview / spec, UI surfacesshape-style design direction: register, scene sentence, named anchor references, anti-references, statesQuestion families absorbed into our interview's UX coverage; its visual-direction probes = our prototype probe
Verification tail, UI surfacescritique (Nielsen rubric 0–4, P0–P3, score trend) + audit (5 measurable dimensions, AI-slop pass/fail)Invoked by our implement/verify as scored gates; findings route into the fix loop

Craft rules (the 7k lines of typography/color/motion judgment, per-model tells, absolute bans) stay external and current — declared external requirement, consent-gated, recorded in external-dependencies.lock.json. bare-minimum-ux remains your five-rule personal overlay on top, always loaded for UI work. Never merged — a fork goes stale in a month.

Installing it: setup-asher-skills owns the moment

Since setup already means "install the selected skill set and run each skill's setup to bind playbooks," a near-always external like Impeccable rides the same flow rather than being a manual afterthought:

  • Offer at detect. Setup sees a UI surface (or you name one) → offers Impeccable as a declared external requirement → provenance review + your consent → recorded, version-pinned, in external-dependencies.lock.json.
  • Install via its own provider (its npm installer / plugin mechanism — we don't vendor or wrap it), then run its setup like any sibling's: /impeccable init writes PRODUCT.md + DESIGN.md + live config as part of project onboarding.
  • Two boundaries. (1) init is an interview — Impeccable itself forbids synthesizing PRODUCT.md from a one-line prompt — so it runs with you present; if setup is running unattended it records a pending step ("run /impeccable init") instead of fabricating context. (2) Setup never wraps Impeccable's own updater; audit mode just diffs recorded version vs installed and re-offers init when PRODUCT.md is missing.
  • Either way, setup guarantees the artifacts' existence path: with Impeccable installed, init is the writer of PRODUCT.md/DESIGN.md; without it, the first interview writes them. Downstream skills read the files, never care who wrote them.

8 · Wayfinder-shaped work — deferred

Agreed: with PRODUCT.md/DESIGN.md, CONTEXT.md/ADRs, and a good interview in place, most "too foggy" cases collapse into talk, then spec — the stack removes the fog that made a map feel necessary. So no new skill now. Worth keeping on the shelf from the pattern, for the day an initiative genuinely outgrows one conversation:

  • The map as index, not store — gist + link, a decision lives in exactly one place;
  • Fog-of-war test — "ticket when you can state the question precisely, fog when you can't; don't pre-slice fog";
  • Out-of-scope as a first-class list that never silently re-enters.

If/when adopted, it needs a name that says what it is — candidates: chart, scout, horizon. Parked.
